Chapter 1

특정 조건 대기: 무식한 스핀 대기(Spin-waiting)의 한계

동시성 세계에서는 스레드가 다음 단계로 가기 전, 특정 상태가 무조건 '참'이 되기를 대기해야 하는 비즈니스 논리가 수없이 많습니다. 예컨대 자식 스레드가 수행을 마쳤는지 확인하고 진행하려는 부모 스레드의 경우가 그렇습니다.

💡 개념 정의

스핀 대기(Spin-waiting): 조건이 만족될 때까지 공유 변수를 무한히 루프 돌며 검사하며 대기하는 방식.

⚙️ 상세 설명 & 작동 원리

스레드는 volatile int done = 0; 과 같은 플래그 변수를 공유하며, 대기하고자 하는 측(부모)은 while (done == 0); 과 같이 세미콜론 하나로 구성된 무한 루프에서 머무릅니다. 자식 스레드가 동작을 마친 뒤 done = 1;로 바꿔주면 루프를 탈출하게 됩니다.

이 방식은 컴파일러 최적화 방지(volatile) 등으로 겨우 동작은 할 수 있지만, CPU 스케줄러 상에서 자신에게 할당된 모든 시간 할당량(Time Slice)을 아무 일도 하지 않는 빈 조건 체크 루프를 도는 데 허비하여 CPU 이용률을 100%로 치솟게 만들어 심각한 연산 자원 낭비를 유발합니다.

🏃 직관적 비유 & 예시

택배가 오늘 밤에 오기로 되어 있는 상황에서, 다른 일을 전혀 하지 않은 채 현관문 앞에서 1초에 한 번씩 문고리를 돌려보며 밖을 쳐다보는 행동과 같습니다. 잠을 자거나 독서를 하면서 편하게 있을 수 있는데도 말이죠!

📌 TL;DR 핵심 요약

단순 루프로 대기하는 것(Spinning)은 CPU 사이클을 가차 없이 버리고 전력을 낭비하며, 스케줄링 시나리오에 따라 오동작을 일으킬 수 있는 동시성 프로그래밍 최대의 악습입니다.

❓ 왜 무한 루프 대기가 논리적으로 틀릴 수도 있나요?

만약 단일 코어 시스템에서 대기하는 부모 스레드가 우선순위가 높다면, 스케줄러가 부모에게 계속 CPU를 배정할 수 있습니다. 부모는 자식이 done = 1을 해주길 기다리며 스핀하지만, 자식은 CPU를 배정받지 못해 done = 1을 영원히 쓰지 못하는 기아 현상(Starvation) 또는 우선순위 역전 문제가 발생해 시스템이 그대로 정지해 버릴 수 있습니다.

Chapter 2

구원의 열쇠: 조건 변수(Condition Variable)와 핵심 API

무의미하게 바쁜 대기(Busy waiting)를 극복하기 위해 등장한 시스템 수준의 추상화 도구가 바로 조건 변수(Condition Variable)입니다.

💡 개념 정의

조건 변수(Condition Variable): 스레드가 원하는 상태(Condition)가 아닐 때 스스로를 휴면 상태로 집어넣을 수 있는 명시적 스레드 대기 큐(Explicit Queue)입니다.

⚙️ 상세 설명 & 작동 원리

조건 변수는 POSIX 표준에서 pthread_cond_t형태로 구현되며 아래 두 가지 핵심 동작을 수행합니다.

pthread_cond_wait(&c, &m);

스레드가 잠들고자 할 때 호출합니다. 이 함수는 인자로 넘어온 락(Mutex)을 원자적으로(Atomically) 완전히 해제하면서 해당 대기 큐에 잠이 듭니다. 이후 나중에 시그널을 받아 잠에서 깨어날 때는 함수에서 탈출하기 전 무조건 락을 다시 획득(Re-acquire)하도록 설계되어 있습니다.

pthread_cond_signal(&c);

조건 변수 큐 c에서 잠자고 있는 대기 스레드 중 정확히 단 하나(One Thread)를 깨워 실행 준비 상태(Ready)로 전환합니다. 만약 큐에 대기 중인 스레드가 없다면 아무 동작도 하지 않고 넘어갑니다.

🏃 직관적 비유 & 예시

은행 대기표 기계와 같습니다. 번호표를 뽑고 대기석에 편안하게 앉아서 졸거나 스마트폰을 하다가(Sleep), 은행원 창구에서 번호를 호출해 띵동 소리를 울리면(Signal) 비로소 일어나 일을 보러 가는 것입니다.

📌 TL;DR 핵심 요약

조건 변수는 자원을 소모하지 않고 안전하게 스레드를 재우는 wait()와, 외부에서 대상을 정확히 찍어 기상시키는 signal()을 매핑하여 바쁜 대기 문제를 완벽히 소거합니다.

Chapter 3

올바른 구현의 삼위일체: 왜 상태 변수(State Var)와 뮤텍스 락(Lock)이 함께 있어야 하는가?

스레드 락과 상태 변수, 그리고 조건 변수 중 단 하나라도 누락되면 즉시 데드락이나 영원한 실종 잠장(Lost Wake-up)의 늪에 빠지게 됩니다.

💥 상태 변수(done)가 없는 경우의 파국

스레드의 완료 플래그(done) 체크 없이 순수하게 wait()와 signal()만 호출하도록 짠다면: 

// [부모 스레드]
pthread_mutex_lock(&m);
pthread_cond_wait(&c, &m); // 대기
pthread_mutex_unlock(&m);

// [자식 스레드]
pthread_mutex_lock(&m);
pthread_cond_signal(&c);   // 기상 시그널
pthread_mutex_unlock(&m);
레이스 컨디션 시나리오: 자식이 부모가 대기 큐에 도달하기도 전에 먼저 실행되어 signal()을 쏘아보내 버리면, 시그널은 공중분해됩니다. 이후 부모가 뒤늦게 wait()에 진입하면 자기는 깨워줄 사람이 없으므로 영원히 동결(Hang) 상태에 봉착합니다.

💥 뮤텍스 락(Mutex Lock)이 없는 경우의 파국

동시성에 락이 배제되어 상태 변수를 안전하게 보존하지 못한 채 접근한다면: 

// [부모 스레드]
if (done == 0) {
    // <-- 이 찰나에 스케줄러가 자식을 끼워넣는다면?
    pthread_cond_wait(&c);
}

// [자식 스레드]
done = 1;
pthread_cond_signal(&c);
레이스 컨디션 시나리오: 부모가 done == 0인 것을 확인하고 잠에 빠기 바로 직전, 자식이 치고 나가 done = 1; signal();을 쏘고 종료됩니다. 직후 부모 스레드는 wait()에 빠지며 자식 스레드 종료 이후에 잠들어 버렸으므로 평생 잠에서 깨어나지 못하는 무덤에 묻힙니다.
💡 개념 정의

조건 동기화의 3요소: 상호배제를 지키는 뮤텍스(Mutex), 프로그램의 구체적 상태를 나타내는 상태 변수(done), 신호를 관리하는 조건 변수(c)가 반드시 연동되어 동작하는 법칙.

⚙️ 상세 설명 & 작동 원리

스레드가 대기하기 전 락을 쥔 상태에서 반드시 상태 변수가 대기 조건에 맞는지 확인하는 구조(while(done == 0) wait(&c, &m);)가 필수적입니다. 이로써 자식이 이미 일을 다 처리했는지(done == 1)를 선제 확인하고, 아직 처리를 하지 않은 상황에서만 안전하게 락을 넘기며 큐로 들어가 대기하게 됩니다.

📌 TL;DR 핵심 요약

조건 변수는 독자적인 상태 저장을 하지 못하는 비상태(Stateless) 도구이므로, 락으로 래핑된 실제 코드 변수값 상태를 함께 비교하는 장치가 있어야만 잠들고 깨어나는 시그널 소실(Lost Wake-up)을 완벽 차단할 수 있습니다.

Chapter 4

생산자-소비자 문제(Bounded Buffer)와 IF 문의 참사

실무의 가장 정통적인 패턴인 유한 버퍼 생산자-소비자(Producer/Consumer)의 1차 버전을 살펴봅시다. 데이터를 밀어넣는 생산자(grep, 웹 수신기)와 데이터를 빼가서 일하는 소비자(wc, 워커 스레드)가 제한된 크기의 버퍼(Buffer)를 두고 공유하는 상황입니다.

버퍼와 if문을 쓴 미완성 코드

// [생산자 - IF 방식의 취약 코드]
void *producer(void *arg) {
    for (int i = 0; i < loops; i++) {
        pthread_mutex_lock(&mutex);
        if (count == 1) { // 버퍼가 차있으면 대기
            pthread_cond_wait(&cond, &mutex);
        }
        put(i); // 버퍼 채우기 (count = 1)
        pthread_cond_signal(&cond); // 깨우기
        pthread_mutex_unlock(&mutex);
    }
}
// [소비자 - IF 방식의 취약 코드]
void *consumer(void *arg) {
    for (int i = 0; i < loops; i++) {
        pthread_mutex_lock(&mutex);
        if (count == 0) { // 버퍼가 비어있으면 대기
            pthread_cond_wait(&cond, &mutex);
        }
        int tmp = get(); // 버퍼 비우기 (count = 0)
        pthread_cond_signal(&cond); // 깨우기
        pthread_mutex_unlock(&mutex);
    }
}

왜 이 코드가 2명의 소비자 환경에서 무조건 터질까요? (Mesa Semantics)

스레드가 깨어났을 때, 내가 대기하던 조건이 '지금 이 순간에도 만족하고 있는지'는 보장되지 않습니다. 깨어난 순간과 락을 다시 확보하고 실행되는 순간 사이에 시간 차이가 발생하기 때문입니다.

단계 스레드 Tc1 (소비자) 스레드 Tc2 (소비자) 스레드 Tp (생산자) 버퍼 상태 (count) 추적 상세 내역 (Mesa Semantics Bug)
1 Running -> wait() 수면 Ready Ready 0 버퍼가 비어있으므로 소비자 Tc1이 락을 놓고 깊은 잠에 빠집니다.
2 Sleep Ready Running (put & signal) 1 생산자 Tp가 버퍼를 1로 만들고 시그널을 발송해 Tc1이 깨어납니다(Ready).
3 Ready Running (get) Sleep / Ready 0 [새치기] 깨어난 Tc1이 락을 잡기 전, 스케줄러가 Tc2를 골라 Tc2가 버퍼의 값을 채갑니다.
4 Running (get) Ready Ready 0 Tc1이 깨어나 if 대기에서 빠져나와 get()을 강행하지만 버퍼는 이미 0입니다. Assertion Fail 발생!
💡 개념 정의

메사 시맨틱(Mesa Semantics): 조건 변수에서 깨어난 스레드가 즉시 실행 상태로 가 자원을 선점하는 것이 아니라 단순히 '준비(Ready)' 상태로만 가기 때문에, 실제 실행될 땐 다른 스레드에 의해 조건이 깨졌을 수 있다는 성질.

⚙️ 상세 설명 & 작동 원리

이에 대비되는 호어 시맨틱(Hoare Semantics)은 깨어난 스레드가 락을 절대적으로 이어받아 강제 실행하도록 보장하지만, 현대 OS 대다수(pthreads 포함)는 효율성을 위해 메사 설계를 차용합니다. 따라서 if 조건 대기는 절대 신뢰할 수 없습니다.

📌 TL;DR 핵심 요약

메사 시맨틱 상에서는 깨어났을 때 누군가 이미 자원을 선점했을 가능성(State 변경)이 다분하므로, 조건 판단 구문은 절대로 if가 아닌 while로 설계하여 반복 재검증해야 합니다.

Chapter 5

진화하는 동기화: While 도입과 '단일 조건 변수'의 한계로 인한 Total Sleep 데드락

Mesa 문제를 회피하기 위해 모든 조건을 while로 바꿨다고 안도하기엔 일찍입니다. 스레드가 3개 이상일 때 단일 조건 변수만 쓰면 또 다른 괴물인 전체 휴면 상태(Total Sleep Deadlock)를 유발하게 됩니다.

💡 개념 정의

전체 휴면(Total Sleep) 데드락: 하나의 조건 변수만 쓰면서 스레드가 다른 성격의 주체를 깨워야 하는데 같은 주체를 깨워, 결국 모든 러너가 서로 자기를 깨워주기만을 기다리며 잠들게 되는 치명적 교착 상태.

⚙️ 상세 설명 & 작동 원리

소비자 T_{c1}, T_{c2}가 대기 중에 생산자 T_p가 버퍼를 채우고 시그널을 발송해 T_{c1}이 일어납니다. T_{c1}이 데이터를 다 소비하고 낸 signal()은 생산자(T_p)를 깨워 다시 채우라고 보내는 의도의 시그널이어야 합니다. 하지만 단일 변수를 쓰다 보니 대기 큐 안의 또 다른 소비자(T_{c2})가 신호를 오해해 깨어나 버립니다.

깨어난 T_{c2}는 버퍼를 보니 이미 T_{c1}이 다 먹어치워 비어있으므로 도로 잠들고, 원래 신호를 받아야 했던 생산자 T_p도 잠자고 있어 누구도 동작하지 않는 "전원 Sleep 상태"가 벌어집니다.

📌 구원의 마스터코드: 두 개의 조건 변수 (empty, fill)

생산자가 기다릴 큐 empty와 소비자가 기다릴 큐 fill을 명시적으로 분리하여 시그널을 각자 정확하게 겨냥해 쏩니다.

// 생산자 - 대기 큐 분리
while (count == 1)
    pthread_cond_wait(&empty, &mutex);
put(i);
pthread_cond_signal(&fill); // 소비자만 겨냥
// 소비자 - 대기 큐 분리
while (count == 0)
    pthread_cond_wait(&fill, &mutex);
int tmp = get();
pthread_cond_signal(&empty); // 생산자만 겨냥

다중 슬롯 원형 버퍼 확장 & 커버링 조건(Covering Conditions)

단일 슬롯을 넘어 다중 슬롯 원형 버퍼(Circular Buffer)로 전환하여 효율성과 병렬 연산 처리량을 대폭 늘릴 수 있습니다. 또한 어떤 스레드가 조건을 만족해 깨어나야 할지 명확하지 않은 가변 연산 환경의 최종 대안인 커버링 조건(Covering Condition)을 정리합니다.

🔄 다중 슬롯 원형 버퍼 (MAX 사이즈 확장)

나머지 연산자(%)를 활용해 버퍼 포인터를 도는 구조입니다. 꽉 차는 임계점 조건이 count == MAX로, 비는 조건이 count == 0으로 정립됩니다. 

void put(int value) {
    buffer[fill_ptr] = value;
    fill_ptr = (fill_ptr + 1) % MAX;
    count++;
}
int get() {
    int tmp = buffer[use_ptr];
    use_ptr = (use_ptr + 1) % MAX;
    count--;
    return tmp;
}

📢 커버링 조건(Covering)과 Broadcast

스레드들 마다 요구하는 메모리 할당 크기가 각각 상이할 때(allocate 100바이트 vs allocate 10바이트) 유용합니다.

메모리가 50바이트 반환되었을 때, signal()만 쏘아보내면 마침 100바이트를 고집하는 대형 스레드 A가 일어나 자기가 처리할 수 없음을 보고 다시 잠듭니다. 하지만 실제로 10바이트만 있으면 잘 달릴 수 있는 소형 스레드 B는 시그널을 받지 못해 계속 깨어나지 못하게 되는 불상사가 일어납니다.

해결 메커니즘 (pthread_cond_broadcast): 락을 풀며 잠자고 있던 모든 스레드를 일괄 깨워(Broadcast) 자원량 조건들을 다시 맞춤 비교 평가하게 합니다. 자원 낭비는 좀 생길 수 있어도 완벽한 비즈니스 정확성을 보증하는 가장 보수적이고 안전한 "커버링 조건" 기법입니다.

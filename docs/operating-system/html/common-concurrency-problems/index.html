0. From semaphores to real concurrency bugs

This note continues from the synchronization tools introduced earlier. Semaphores provide a general mechanism for coordinating threads, but real programs still fail when programmers protect the wrong region, forget a required ordering, or acquire locks in an unsafe pattern.

Lock

A binary semaphore can implement mutual exclusion: only one thread enters a critical section.

Ordering

A semaphore can force B to wait until A has completed.

Producer/Consumer

Empty/full counters plus a mutex coordinate bounded-buffer access.

Reader/Writer

Multiple readers may share access, while writers need exclusive access.

The central question is: if locks, condition variables, and semaphores exist, why do concurrency bugs still appear in large applications? The answer is that synchronization is not only about using a primitive; it is about matching the primitive to the guarantee the program needs.

1. The main classification

Start by deciding whether the problem is a wrong value, a wrong order, or no progress.

Atomicity violation

The program expected a group of operations to be indivisible, but another thread interleaved inside it.

Order violation

The program expected A before B, but the scheduler allowed B before A.

Deadlock

Threads wait forever because each one needs a resource held by another thread in a cycle.

2. Non-deadlock bug: atomicity violation

Definition

An atomicity violation occurs when multiple memory accesses were intended to behave as one serializable unit, but that atomicity was not enforced during execution. The bug is not simply “two threads touched the same variable”; the bug is that a region that should have been one indivisible action was split by an interleaving.

  • Intention: the whole code region should be atomic.
  • Reality: without a common lock discipline, the scheduler may run another thread in the middle.
  • Result: check-then-use errors, stale reads, lost updates, or inconsistent state.

Canonical check-then-use failure

One thread checks whether a pointer is non-null and later uses it. Another thread can set the same pointer to null after the check but before the use.

// Thread 1
if (proc_info) {
    ...
    fputs(proc_info, ...);
    ...
}

// Thread 2
proc_info = NULL;
  1. Thread 1 checks proc_info and sees a non-null value.
  2. Thread 1 is interrupted before the call that uses the pointer.
  3. Thread 2 runs and sets proc_info = NULL.
  4. Thread 1 resumes and uses a value that is no longer valid.

Fix: protect all related accesses with the same lock

The check, the use, and the write that changes the pointer must follow the same lock discipline. Adding a lock in only one place is not enough; all related accesses to the shared state must agree on the same protection rule.

pthread_mutex_t proc_info_lock = PTHREAD_MUTEX_INITIALIZER;

// Thread 1
pthread_mutex_lock(&proc_info_lock);
if (proc_info) {
    ...
    fputs(proc_info, ...);
    ...
}
pthread_mutex_unlock(&proc_info_lock);

// Thread 2
pthread_mutex_lock(&proc_info_lock);
proc_info = NULL;
pthread_mutex_unlock(&proc_info_lock);

Key lesson: an atomicity bug is fixed by identifying the logical operation that must be atomic, then ensuring every access participating in that operation is protected consistently.

3. Non-deadlock bug: order violation

Definition

An order violation occurs when two memory accesses or two groups of operations must happen in a particular order, but that order is not enforced. In other words, A must always happen before B, but the execution allows B to happen first.

  • Intention: initialization must complete before use.
  • Reality: a newly created thread may run immediately, before the creator finishes setting fields.
  • Result: use-before-initialization, null pointer dereference, or reading an invalid state.

Thread creation does not imply initialization order

Suppose one thread creates another and stores the created thread handle in a shared variable. The worker may run immediately after creation and read the variable before the creator has finished setting it.

// Thread 1
void init() {
    ...
    workerThread = createThread(workerMain, ...);
    ...
}

// Thread 2
void workerMain(...) {
    ...
    state = workerThread->state;
    ...
}

The bug is not primarily a mutual-exclusion problem. The key missing guarantee is a happens-before relationship: initialization must complete before the worker reads the initialized state.

Fix: condition variable and an explicit predicate

Use a mutex, a condition variable, and a predicate such as initialized. The waiting thread does not proceed until the predicate says the required order has been achieved.

// Thread 1
pthread_mutex_lock(&mtLock);
initialized = 1;
pthread_cond_signal(&mtCond);
pthread_mutex_unlock(&mtLock);
// Thread 2
pthread_mutex_lock(&mtLock);
while (initialized == 0)
    pthread_cond_wait(&mtCond, &mtLock);
pthread_mutex_unlock(&mtLock);
state = workerThread->state;

Why a while loop? A condition-variable wait may return even when the condition is not true, and another thread can change the condition before the woken thread runs. Therefore, the predicate must be checked in a loop, not an if.

4. Deadlock bug

Basic lock-order conflict

Two threads can deadlock if they acquire the same locks in opposite orders. Each thread holds one lock and waits for the other lock forever.

// Thread 1
pthread_mutex_lock(L1);
pthread_mutex_lock(L2);

// Thread 2
pthread_mutex_lock(L2);
pthread_mutex_lock(L1);
  1. Thread 1 acquires L1.
  2. Thread 2 acquires L2.
  3. Thread 1 waits for L2.
  4. Thread 2 waits for L1.
  5. Neither can proceed, because each is waiting for the other.

This is a cycle in the wait-for relation, often represented using a resource-allocation graph.

5. Why deadlocks occur in real systems

Large-codebase dependencies

In a large system, one subsystem may call another while holding a lock. A virtual memory subsystem might call a file subsystem to page in data from disk. The file subsystem might later need memory and call back into the virtual memory subsystem. If both sides hold internal locks, a circular dependency can emerge.

VMS→File System→VMS

Encapsulation

Encapsulation hides implementation details, including internal locks. That is good software design, but it can make global lock ordering difficult. For example, a container method that adds all elements from another container may need to lock both the receiver and the argument. Two threads calling the method in opposite directions can acquire those hidden locks in opposite orders.

// Thread 1
v1.addAll(v2);

// Thread 2
v2.addAll(v1);

6. The four necessary conditions

Deadlock can occur only when all four conditions hold simultaneously. Prevention breaks at least one.

Interactive: trylock and livelock

Releasing and retrying can prevent deadlock, but synchronized retries can produce livelock.

T1: lock(L1), trylock(L2)
T2: lock(L2), trylock(L1)

    7. Deadlock prevention strategies

    The practical way to reason about deadlock prevention is to connect each strategy to the deadlock condition it breaks. The answer is not just memorizing the four conditions; it is knowing what design move removes one of them and what trade-off that move creates.

    7.1 Break circular wait: lock ordering

    Define a total or partial ordering over locks and require all threads to acquire locks in that order. If every thread acquires L1 before L2, a two-lock cycle cannot form.

    Trade-off: practical and common, but difficult to enforce across large codebases and hidden library locks.

    7.2 Break hold-and-wait: acquire all at once

    Acquire all required locks during a single protected acquisition phase, so a thread does not hold some locks while waiting for others.

    Trade-off: can reduce concurrency and may require knowing all needed locks in advance.

    7.3 Relax no preemption: trylock

    Use pthread_mutex_trylock(); if the second lock is unavailable, release the first lock and retry instead of waiting while holding it.

    Trade-off: prevents the wait-while-holding pattern but can create livelock and requires careful cleanup of all resources.

    7.4 Reduce mutual exclusion: lock-free design

    Use atomic primitives such as compare-and-swap to update shared state without a traditional mutex.

    Trade-off: not automatically simpler or faster; retry loops, ABA, memory reclamation, and starvation can be hard.

    Acquire-all-at-once pattern

    pthread_mutex_lock(prevention);   // begin lock acquisition
pthread_mutex_lock(L1);
pthread_mutex_lock(L2);
...
pthread_mutex_unlock(prevention); // end acquisition phase

    Trylock pattern and cleanup warning

    top:
pthread_mutex_lock(L1);
if (pthread_mutex_trylock(L2) != 0) {
    pthread_mutex_unlock(L1);
    goto top;
}

    If the code acquired any other resource before jumping back to top, it must release that resource too: other locks, memory allocations, file handles, or temporary state. Otherwise the retry loop may create leaks or new synchronization bugs.

    8. Lock-free approach and compare-and-swap

    Lock-free techniques attempt to reduce dependence on mutual exclusion. The central primitive here is Compare-And-Swap (CAS): update a memory location only if it still contains the value you expected.

    int CompareAndSwap(int *address, int expected, int new_value) {
    if (*address == expected) {
        *address = new_value;
        return 1; // success
    }
    return 0;     // failure
}

    Atomic increment

    The thread repeatedly reads the old value and tries to replace it with old + amount. If another thread changed the value first, CAS fails and the loop retries with a fresh old value.

    void AtomicIncrement(int *value, int amount) {
    do {
        int old = *value;
    } while (CompareAndSwap(value, old, old + amount) == 0);
}

    List insertion: from race to lock to CAS

    A naive linked-list insert has a race because multiple threads can read the same head and overwrite one another’s update.

    Race-prone
    void insert(int value) {
    node_t *n = malloc(sizeof(node_t));
    n->value = value;
    n->next = head;
    head = n;
}
    Mutex-based
    void insert(int value) {
    node_t *n = malloc(sizeof(node_t));
    n->value = value;
    pthread_mutex_lock(listlock);
    n->next = head;
    head = n;
    pthread_mutex_unlock(listlock);
}
    CAS-based
    void insert(int value) {
    node_t *n = malloc(sizeof(node_t));
    n->value = value;
    do {
        n->next = head;
    } while (CompareAndSwap(&head, n->next, n) == 0);
}

    The CAS version succeeds only if head still equals the value stored in n->next. If another thread changed head, the CAS fails, the loop refreshes n->next, and the thread tries again.

    Important caveat: lock-free does not mean “free.” It can introduce ABA problems, memory-reclamation difficulties, retry overhead, and starvation. For this note, the key idea is that CAS weakens the need for mutual exclusion, thereby attacking one condition that enables deadlock.

    9. Deadlock avoidance via scheduling

    If the system knows in advance which locks each thread may need, it can avoid scheduling dangerous combinations at the same time. For example:

    ThreadNeeds L1?Needs L2?
    T1YesYes
    T2YesYes
    T3NoYes
    T4NoNo

    This approach is only useful in limited environments, such as embedded systems, because general-purpose systems rarely know every future lock requirement in advance.

    10. Detect and recover

    Another strategy is to allow deadlocks to occur occasionally, detect them, and recover. This is reasonable when deadlocks are rare and the recovery cost is lower than complex prevention.

    • Kill or restart a stuck thread/process.
    • Roll back a transaction.
    • Restart a service or subsystem.

    The blunt recovery strategy is simply: restart. It is not elegant, but it can be economically rational when failure is rare and restart is cheap.

    12. Likely exam questions

    1. Define atomicity violation and order violation, then explain the difference with an interleaving example.
    2. Why is the pattern if (ptr) use(ptr) dangerous in a multithreaded program?
    3. When fixing an order violation with a condition variable, why should the wait be wrapped in a while loop?
    4. List the four necessary conditions for deadlock and connect each one to a prevention technique.
    5. Explain how lock ordering breaks circular wait and why it is hard to enforce in a large codebase.
    6. Why can a pthread_mutex_trylock()-based solution produce livelock, and how can random delay help?
    7. Compare mutex-based list insertion with CAS-based lock-free insertion.
    8. Why is deadlock avoidance via scheduling more realistic in embedded systems than in general-purpose operating systems?

    13. Common pitfalls

    • Calling every atomicity violation just a “race condition” without explaining the missing atomic region.
    • Trying to fix an order violation only with mutual exclusion, when the real missing guarantee is ordering.
    • Memorizing the four deadlock conditions without connecting them to prevention strategies.
    • Assuming trylock automatically solves everything; livelock and cleanup problems remain.
    • Assuming lock-free code is always faster or safer; CAS loops and memory management are difficult.

    Quick self-check

    14. One-sentence summary

    Common concurrency bugs can be organized as atomicity violations, order violations, and deadlocks; the corresponding fixes are to make the right region atomic, establish the necessary happens-before relation, or break at least one of the deadlock conditions.

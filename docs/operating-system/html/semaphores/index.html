Core Explanation

0. 이 노트에서 반드시 잡아야 하는 큰 그림

이 노트의 핵심은 semaphore가 단순히 lock의 다른 이름이 아니라, 정수값을 이용해 여러 종류의 동기화 문제를 표현하는 일반적인 primitive라는 점이다. 이전 노트에서 다룬 lock과 condition variable은 각각 역할이 비교적 분명했다.

  • Lock: critical section에 동시에 하나의 thread만 들어가게 한다.
  • Condition variable: 어떤 조건이 참이 될 때까지 thread를 잠재웠다가, 조건을 바꾼 thread가 깨운다.

Semaphore는 이 둘을 모두 흉내낼 수 있다. 단, 이것이 장점이면서 동시에 위험한 점이다. semaphore 하나로 여러 의미를 표현할 수 있기 때문에, 코드만 보고도 “이 semaphore는 lock 역할인가, event ordering 역할인가, resource count 역할인가?”를 구분해야 한다.

핵심 사고법

Semaphore의 value는 “지금 당장 통과 가능한 허가증(token)의 개수”라고 생각하면 좋다. sem_wait()는 허가증 하나를 가져가는 연산이고, 허가증이 없으면 기다린다. sem_post()는 허가증 하나를 돌려주는 연산이며, 기다리는 thread가 있으면 하나를 깨울 수 있다.

1. 지난 시간 복습: Condition Variable과 Join 패턴

첫 예제는 condition variable을 이용해 parent thread가 child thread의 종료를 기다리는 패턴을 다시 보여준다.

void thr_exit() {
    pthread_mutex_lock(&m);
    done = 1;
    pthread_cond_signal(&c);
    pthread_mutex_unlock(&m);
}

void thr_join() {
    pthread_mutex_lock(&m);
    while (done == 0)
        pthread_cond_wait(&c, &m);
    pthread_mutex_unlock(&m);
}

여기서 핵심은 done이라는 공유 변수가 condition이고, mutex m이 그 condition을 보호한다는 점이다.

왜 while (done == 0)인가?

pthread_cond_wait()에서 깨어났다고 해서 곧바로 조건이 참이라고 보장할 수 없다. 이유는 여러 가지다.

  1. Spurious wakeup: 실제 signal이 없었는데도 깨어날 수 있다.
  2. 경쟁 상황: signal을 받은 뒤 lock을 다시 얻기 전에 다른 thread가 상태를 바꿀 수 있다.
  3. 여러 대기자: signal/broadcast 후 깨어난 thread 중 일부만 조건을 만족할 수 있다.

따라서 condition variable 패턴에서는 항상 if보다 while을 사용해 조건을 다시 확인하는 것이 안전하다.

Semaphore와의 연결

이 노트에서는 이 join 패턴을 semaphore로 훨씬 간단하게 표현할 수 있음을 보게 된다. 예를 들어 child가 끝나기 전까지 parent를 기다리게 하고 싶다면, 초기값 0인 semaphore를 두고 parent는 sem_wait(), child는 sem_post()를 호출하면 된다.

2. POSIX Semaphore API의 의미

API 예제는 POSIX semaphore의 기본 API를 소개한다.

int sem_init(sem_t *s, int pshared, unsigned int value);
int sem_wait(sem_t *s);
int sem_post(sem_t *s);

sem_init()

sem_init(&s, 0, value)는 semaphore s를 초기값 value로 초기화한다.

  • pshared = 0: 같은 process 내부의 thread들이 semaphore를 공유한다.
  • pshared != 0: 서로 다른 process들이 semaphore를 공유한다. 이 경우 semaphore 객체는 shared memory 영역에 있어야 한다.

OS 수업에서 대부분의 예제는 pthread 기반 thread 동기화이므로 pshared = 0을 사용한다.

sem_wait()

sem_wait()는 전통적으로 P 연산 또는 down 연산이라고 부른다. 직관적으로는 다음과 같다.

  1. semaphore value가 양수이면 value를 1 감소시키고 통과한다.
  2. value가 0이면 통과할 token이 없으므로 sleep/block 상태로 들어간다.
  3. 나중에 다른 thread가 sem_post()를 호출하면 깨어나 다시 진행한다.

즉 sem_wait()는 “자원을 하나 확보하고 들어가기”에 해당한다.

sem_post()

sem_post()는 V 연산 또는 up 연산이라고 부른다.

  1. semaphore value를 1 증가시킨다.
  2. 기다리는 thread가 있으면 그중 하나를 깨운다.

즉 sem_post()는 “자원을 하나 반환하거나 event 발생을 알리기”에 해당한다.

“negative value” 설명

일부 설명에서는 sem_wait()가 값을 감소시키고, value가 negative이면 기다린다고 설명되어 있다. 이는 고전적인 semaphore 명세에서 자주 쓰는 설명이다. 이 모델에서는 value가 음수일 때 그 절댓값이 기다리는 thread 수를 나타낼 수 있다.

하지만 실제 Linux 구현에서는 value를 0 아래로 내리지 않고, 대기자 수는 별도의 wait queue로 관리할 수 있다. 시험에서는 보통 다음처럼 정리하면 안전하다.

정리

개념적으로 semaphore value는 사용 가능한 resource/token 수를 나타낸다. 구현에 따라 value가 음수가 되어 waiting thread 수를 반영한다고 설명할 수도 있고, Linux처럼 value는 0 미만으로 내려가지 않고 대기 큐를 따로 관리할 수도 있다.

3. Binary Semaphore: Lock으로 쓰는 경우

binary semaphore 예제는 semaphore를 lock처럼 사용하는 예시이다.

sem_t m;
sem_init(&m, 0, 1);

sem_wait(&m);
// critical section here
sem_post(&m);

초기값이 1이라는 것은 “critical section에 들어갈 수 있는 허가증이 하나 있다”는 뜻이다.

실행 흐름

  1. 처음 value는 1이다.
  2. Thread A가 sem_wait(&m)을 호출한다.
  3. value가 1이므로 A는 통과하고 value는 0이 된다.
  4. Thread A가 critical section을 실행한다.
  5. 그 사이 Thread B가 sem_wait(&m)을 호출하면 value가 0이므로 기다린다.
  6. A가 critical section을 끝내고 sem_post(&m)을 호출한다.
  7. value가 증가하고, 기다리던 B가 깨어나 critical section에 들어간다.

왜 binary semaphore인가?

value가 실질적으로 0 또는 1 상태만 오가기 때문이다.

  • 1: 아무도 critical section에 없으므로 들어갈 수 있다.
  • 0: 누군가 critical section에 있으므로 기다려야 한다.

Lock과 완전히 같은가?

용도는 비슷하지만 완전히 같은 개념은 아니다. 일반적인 mutex lock은 “lock을 잡은 thread가 unlock해야 한다”는 ownership 개념이 강하다. 반면 semaphore는 한 thread가 wait하고 다른 thread가 post하는 방식도 자연스럽게 가능하다. 이 차이 때문에 semaphore는 lock뿐 아니라 ordering에도 사용할 수 있다.

시험 함정

“Semaphore는 lock이다”라고 쓰면 부족하다. 정확히는 “초기값 1인 semaphore를 critical section 앞뒤에 wait/post로 배치하면 binary lock처럼 mutual exclusion을 제공한다”라고 써야 한다.

4. Semaphore로 Ordering 표현하기

ordering 예제는 semaphore를 condition variable처럼 사용하는 예제이다.

sem_t s;

void *child(void *arg) {
    printf("child\n");
    sem_post(&s);
    return NULL;
}

int main(int argc, char *argv[]) {
    pthread_t c;
    sem_init(&s, 0, X); // what should X be?
    printf("parent: begin\n");
    pthread_create(&c, NULL, child, NULL);
    sem_wait(&s);
    printf("parent: end\n");
    return 0;
}

목표는 출력 순서를 다음처럼 강제하는 것이다.

parent: begin
child
parent: end

X는 왜 0이어야 하는가?

sem_init(&s, 0, 0)이면 처음에는 통과 가능한 token이 없다. parent는 child를 만든 뒤 sem_wait(&s)를 호출하지만, value가 0이므로 기다린다. child가 실행되어 child를 출력하고 sem_post(&s)를 호출하면 token이 하나 생기고 parent가 깨어난다. 그 뒤에야 parent: end가 출력된다.

반면 X = 1이면 parent는 sem_wait(&s)에서 바로 통과할 수 있다. 그러면 child가 아직 실행되지 않았는데도 parent: end가 먼저 출력될 수 있다. 따라서 ordering 보장이 깨진다.

Condition variable과 비교

condition variable로 같은 일을 하려면 done 변수, mutex, condvar가 필요했다. semaphore는 초기값 0 자체가 “아직 event가 발생하지 않음”을 표현하므로 코드가 더 짧아진다.

하지만 condition variable은 명시적인 predicate, 예를 들어 done == 1, count > 0 같은 조건을 함께 두는 스타일이다. semaphore는 value 안에 조건 상태를 압축해서 넣는다. 그래서 간단한 ordering에는 편하지만, 복잡한 조건에서는 condition variable이 더 명확할 때도 있다.

5. Producer/Consumer Problem의 본질

bounded buffer 기반 Producer/Consumer 문제를 다룬다. 이 문제는 OS 동시성에서 매우 중요하다.

  • Producer는 buffer에 item을 넣는다.
  • Consumer는 buffer에서 item을 꺼낸다.
  • Buffer 크기는 MAX로 제한되어 있다.

따라서 두 종류의 조건이 생긴다.

  1. Producer는 buffer가 가득 차 있으면 기다려야 한다.
  2. Consumer는 buffer가 비어 있으면 기다려야 한다.

또한 buffer 자체는 공유 자료구조이므로, 여러 producer/consumer가 동시에 fill, use, buffer를 건드리면 race condition이 생긴다.

즉 Producer/Consumer 문제에는 두 문제가 동시에 존재한다.

  • Condition synchronization: empty/full 조건을 기다리는 문제
  • Mutual exclusion: buffer 자료구조를 안전하게 조작하는 문제

이 둘을 정확히 분리하는 것이 이 노트의 핵심이다.

6. Bounded Buffer 구조

bounded-buffer 코드는 circular buffer이다.

int buffer[MAX]; // bounded buffer
int fill = 0;
int use = 0;

void put(int value) {
    buffer[fill] = value;
    fill = (fill + 1) % MAX;
}

int get() {
    int tmp = buffer[use];
    use = (use + 1) % MAX;
    return tmp;
}

fill과 use

  • fill: 다음 producer가 item을 넣을 index
  • use: 다음 consumer가 item을 꺼낼 index

(index + 1) % MAX를 사용하는 이유는 buffer를 원형 큐처럼 쓰기 위해서이다. 예를 들어 MAX = 5이면 index는 0, 1, 2, 3, 4 다음 다시 0으로 돌아간다.

왜 공유 상태인가?

buffer, fill, use는 모든 producer와 consumer가 함께 접근한다. 따라서 두 producer가 동시에 fill을 읽거나, 두 consumer가 동시에 use를 읽으면 문제가 생길 수 있다.

예를 들어 fill = 2일 때 producer P1과 P2가 동시에 put()에 들어가면 다음 일이 가능하다.

  1. P1이 fill = 2를 읽는다.
  2. P2도 fill = 2를 읽는다.
  3. P1이 buffer[2]에 값을 쓴다.
  4. P2가 buffer[2]에 다른 값을 덮어쓴다.
  5. 둘 다 fill = 3으로 갱신한다.

결과적으로 item 하나가 사라지고 buffer 상태가 꼬인다. 이것이 race condition이다.

7. empty/full Semaphore의 의미

초기 producer/consumer 예제는 empty와 full semaphore를 도입한다.

sem_t empty, full;

sem_init(&empty, 0, MAX); // MAX are empty
sem_init(&full, 0, 0);    // 0 are full

empty

empty는 buffer 안의 빈 slot 수를 나타낸다. 처음에는 buffer가 완전히 비어 있으므로 빈 slot이 MAX개이다.

Producer는 item을 넣기 전에 빈 slot 하나를 확보해야 한다.

sem_wait(&empty);
put(i);
sem_post(&full);

sem_wait(&empty)는 빈 slot 하나를 예약하는 의미이다. 만약 빈 slot이 없으면 producer는 기다린다.

full

full은 buffer 안의 채워진 item 수를 나타낸다. 처음에는 item이 없으므로 0이다.

Consumer는 item을 꺼내기 전에 채워진 slot 하나를 확보해야 한다.

sem_wait(&full);
tmp = get();
sem_post(&empty);

sem_wait(&full)은 꺼낼 item 하나를 예약하는 의미이다. 만약 item이 없으면 consumer는 기다린다.

중요한 불변식

정상적인 bounded buffer에서는 항상 empty + full = MAX라는 직관적 관계가 유지된다. producer가 item을 하나 넣으면 empty가 하나 줄고 full이 하나 늘어난다. consumer가 item을 하나 꺼내면 full이 하나 줄고 empty가 하나 늘어난다.

시험에 좋은 표현

empty와 full은 buffer의 물리적 접근을 보호하는 lock이 아니라, buffer의 상태 조건을 표현하는 counting semaphore이다.

8. Single Producer/Consumer에서는 왜 우연히 동작하는가?

race-condition 부분은 single producer와 single consumer에서는 앞의 코드가 동작한다고 말한다. 이유는 fill은 producer만 바꾸고 use는 consumer만 바꾸기 때문이다.

  • Producer 하나만 fill을 증가시킨다.
  • Consumer 하나만 use를 증가시킨다.

따라서 fill끼리의 경쟁, use끼리의 경쟁은 발생하지 않는다. 물론 buffer slot 자체에 대해서는 empty/full이 순서를 맞춰주므로 어느 정도 안전해진다.

하지만 producer가 둘 이상이 되면 여러 producer가 fill을 동시에 조작한다. consumer가 둘 이상이면 여러 consumer가 use를 동시에 조작한다. 따라서 mutual exclusion이 필요해진다.

9. 잘못된 Mutex 추가가 만드는 Deadlock

매우 중요한 함정이다. Race condition을 막기 위해 mutex를 추가해야 한다는 생각은 맞다. 하지만 위치가 잘못되면 deadlock이 생긴다.

문제 있는 구조는 다음과 같다.

// producer
sem_wait(&mutex);
sem_wait(&empty);
put(i);
sem_post(&full);
sem_post(&mutex);

// consumer
sem_wait(&mutex);
sem_wait(&full);
tmp = get();
sem_post(&empty);
sem_post(&mutex);

왜 위험한가?

mutex를 먼저 잡은 뒤 empty 또는 full을 기다린다. 그런데 empty/full의 값이 바뀌려면 반대쪽 thread가 buffer를 조작해야 하고, buffer를 조작하려면 그 thread도 mutex가 필요하다.

구체적 deadlock 시나리오

초기 buffer가 비어 있다고 하자. 즉 full = 0이다.

  1. Consumer C가 먼저 실행된다.
  2. C가 sem_wait(&mutex)로 mutex를 잡는다.
  3. C가 sem_wait(&full)을 호출한다.
  4. full = 0이므로 C는 item이 생길 때까지 기다린다.
  5. 그런데 C는 mutex를 잡은 채로 잠들어 있다.
  6. Producer P가 실행되어 item을 넣으려 한다.
  7. P는 먼저 sem_wait(&mutex)를 해야 하는데, mutex는 C가 잡고 있다.
  8. P는 mutex를 기다린다.
  9. C는 P가 item을 넣고 full을 post하기를 기다린다.
  10. P는 C가 mutex를 놓기를 기다린다.

이제 둘 다 서로를 기다리므로 deadlock이다.

핵심 원인

조건을 만족시킬 수 있는 thread가 필요로 하는 mutex를, 조건을 기다리는 thread가 잡고 놓지 않는 구조가 문제이다.

10. 올바른 Producer/Consumer 해법

올바른 해결책은 empty/full wait를 먼저 하고, 그 다음 mutex를 잡는 것이다.

void *producer(void *arg) {
    int i;
    for (i = 0; i < loops; i++) {
        sem_wait(&empty);
        sem_wait(&mutex);
        put(i);
        sem_post(&mutex);
        sem_post(&full);
    }
}

void *consumer(void *arg) {
    int i;
    for (i = 0; i < loops; i++) {
        sem_wait(&full);
        sem_wait(&mutex);
        int tmp = get();
        sem_post(&mutex);
        sem_post(&empty);
    }
}

Producer의 의미

  1. sem_wait(&empty): 빈 slot 하나를 예약한다. buffer가 가득 차 있으면 여기서 기다린다.
  2. sem_wait(&mutex): buffer 자료구조를 조작하기 위해 lock을 잡는다.
  3. put(i): 예약한 빈 slot에 item을 넣는다.
  4. sem_post(&mutex): buffer 조작이 끝났으므로 lock을 푼다.
  5. sem_post(&full): 채워진 slot이 하나 늘었음을 알린다.

Consumer의 의미

  1. sem_wait(&full): 채워진 slot 하나를 예약한다. buffer가 비어 있으면 여기서 기다린다.
  2. sem_wait(&mutex): buffer 자료구조를 조작하기 위해 lock을 잡는다.
  3. get(): 예약한 item을 꺼낸다.
  4. sem_post(&mutex): buffer 조작이 끝났으므로 lock을 푼다.
  5. sem_post(&empty): 빈 slot이 하나 늘었음을 알린다.

왜 deadlock이 사라지는가?

이제 thread가 empty나 full에서 기다릴 때는 mutex를 잡고 있지 않다. 따라서 반대쪽 thread가 mutex를 얻어 buffer 상태를 바꾸고, 필요한 semaphore를 post할 수 있다.

가장 중요한 한 줄

Producer/Consumer에서 empty/full은 조건 동기화이고, mutex는 buffer 조작 보호이다. 조건을 기다리는 동안 mutex를 잡고 있으면 안 된다.

11. Reader-Writer Lock의 필요성

reader-writer lock의 핵심은 다음과 같다. 일반 mutex는 단순하다. 어떤 자료구조에 접근하려면 무조건 하나의 thread만 들어가게 한다. 하지만 읽기 작업이 많고 쓰기 작업이 적은 자료구조에서는 이것이 비효율적일 수 있다.

예를 들어 linked list나 hash table에서 lookup은 자료구조를 읽기만 한다. 여러 thread가 동시에 lookup을 해도 자료구조가 변하지 않으므로 안전할 수 있다. 반면 insert나 delete는 자료구조를 바꾸므로 다른 접근과 동시에 실행되면 위험하다.

Reader-writer lock은 다음 규칙을 제공한다.

  • Reader끼리는 동시에 들어갈 수 있다.
  • Writer는 혼자만 들어갈 수 있다.
  • Writer가 들어가 있는 동안 reader는 들어갈 수 없다.
  • Reader가 하나라도 들어가 있으면 writer는 들어갈 수 없다.

즉 목표는 many readers or one writer이다.

12. Reader-Writer Lock 자료구조

reader-writer lock 구조체는 다음과 같다.

typedef struct _rwlock_t {
    sem_t lock;
    sem_t writelock;
    int readers;
} rwlock_t;

각 필드의 역할은 매우 중요하다.

readers

현재 read critical section 안에 있는 reader 수이다. reader가 들어올 때 증가하고, 나갈 때 감소한다.

lock

readers 변수 자체를 보호하는 binary semaphore이다. readers++와 readers--는 단순한 한 줄처럼 보이지만 실제로는 load, add, store로 이루어져 race condition이 생길 수 있다. 따라서 readers count를 바꿀 때는 lock이 필요하다.

writelock

writer와 reader group 사이의 큰 문이다.

  • Writer는 이 writelock을 잡아야 들어간다.
  • 첫 번째 reader도 이 writelock을 잡아 writer를 막는다.
  • 마지막 reader가 나갈 때 writelock을 풀어 writer가 들어갈 수 있게 한다.

초기화는 다음과 같다.

void rwlock_init(rwlock_t *rw) {
    rw->readers = 0;
    sem_init(&rw->lock, 0, 1);
    sem_init(&rw->writelock, 0, 1);
}

둘 다 초기값 1인 이유는 처음에는 readers count도 자유롭게 갱신할 수 있고, writer/reader group이 들어갈 수 있는 큰 문도 열려 있기 때문이다.

13. Writer Lock의 동작

writer lock은 단순하다.

void rwlock_acquire_writelock(rwlock_t *rw) {
    sem_wait(&rw->writelock);
}

void rwlock_release_writelock(rwlock_t *rw) {
    sem_post(&rw->writelock);
}

Writer는 자료구조를 변경하므로 단독 접근이 필요하다. 따라서 writelock만 얻으면 된다. 이때 reader group이 이미 들어가 있으면 첫 reader가 writelock을 잡고 있으므로 writer는 기다린다. 다른 writer가 이미 들어가 있어도 writelock이 잡혀 있으므로 기다린다.

14. Reader Lock의 동작

reader lock은 핵심 로직이 들어 있다.

void rwlock_acquire_readlock(rwlock_t *rw) {
    sem_wait(&rw->lock);
    rw->readers++;
    if (rw->readers == 1)
        sem_wait(&rw->writelock);
    sem_post(&rw->lock);
}

첫 번째 reader

첫 번째 reader가 들어오면 readers가 0에서 1이 된다. 이 reader는 reader group을 대표해서 writelock을 잡는다. 이렇게 해야 writer가 동시에 들어오지 못한다.

두 번째 이후 reader

이미 첫 reader가 writelock을 잡고 있으므로, 두 번째 이후 reader는 writelock을 다시 잡을 필요가 없다. 그냥 readers count만 증가시키고 read critical section으로 들어간다. 이 때문에 여러 reader가 동시에 실행될 수 있다.

Release는 다음과 같다.

void rwlock_release_readlock(rwlock_t *rw) {
    sem_wait(&rw->lock);
    rw->readers--;
    if (rw->readers == 0)
        sem_post(&rw->writelock);
    sem_post(&rw->lock);
}

마지막 reader

reader가 나갈 때 readers를 감소시킨다. 만약 감소 후 readers == 0이면 이 reader가 마지막 reader이다. 이제 reader group이 완전히 사라졌으므로 writelock을 풀어 writer가 들어갈 수 있게 한다.

Starvation 문제

이 구현은 reader에게 유리한 구조이다. writer가 기다리고 있어도 새로운 reader가 계속 들어오면 writer가 매우 오래 기다릴 수 있다. 이를 writer starvation이라고 한다.

시험 포인트

Reader-writer lock은 병렬성을 높이지만 공정성 문제가 생길 수 있다. “reader가 많으면 성능이 좋아진다”에서 끝내지 말고, writer starvation trade-off까지 언급해야 좋은 답안이다.

15. Semaphore를 Lock과 Condition Variable로 구현하기

semaphore 자체는 lock과 condition variable로 구현할 수 있다.

typedef struct __Sem_t {
    int value;
    pthread_cond_t cond;
    pthread_mutex_t lock;
} Sem_t;

여기서 value는 사용 가능한 token 수이고, cond는 token이 없을 때 기다리는 thread를 재우는 용도이며, lock은 value와 condition variable 조작을 보호한다.

초기화는 다음과 같다.

void Sem_init(Sem_t *s, int value) {
    s->value = value;
    Cond_init(&s->cond);
    Mutex_init(&s->lock);
}

일부 설명에서는 “only one thread can call this”라고 되어 있다. 초기화 도중 여러 thread가 동시에 접근하면 아직 완성되지 않은 semaphore를 사용할 수 있기 때문이다.

16. Sem_wait() 구현의 의미

void Sem_wait(Sem_t *s) {
    Mutex_lock(&s->lock);
    while (s->value <= 0)
        Cond_wait(&s->cond, &s->lock);
    s->value--;
    Mutex_unlock(&s->lock);
}

단계별 의미

  1. Mutex_lock(&s->lock): value를 안전하게 확인하기 위해 lock을 잡는다.
  2. while (s->value <= 0): 사용 가능한 token이 없으면 기다린다.
  3. Cond_wait(&s->cond, &s->lock): lock을 원자적으로 놓고 sleep한다. 깨어나면 lock을 다시 잡고 돌아온다.
  4. 조건이 만족되면 s->value--로 token 하나를 사용한다.
  5. lock을 푼다.

왜 while인가?

condition variable과 마찬가지로 깨어났다고 해서 token이 반드시 있다고 보장할 수 없다. 여러 thread가 동시에 깨어났거나, signal 후 다른 thread가 먼저 token을 가져갈 수 있다. 따라서 while로 조건을 다시 확인해야 한다.

17. Sem_post() 구현의 의미

void Sem_post(Sem_t *s) {
    Mutex_lock(&s->lock);
    s->value++;
    Cond_signal(&s->cond);
    Mutex_unlock(&s->lock);
}

Sem_post()는 token을 하나 추가하고 기다리는 thread 하나를 깨운다. 중요한 점은 value++와 Cond_signal()이 lock 안에서 이루어진다는 것이다. 그래야 기다리는 thread와 post하는 thread 사이의 race를 줄이고 상태 변화와 알림이 일관되게 유지된다.

원래 명세와 Linux 구현 차이

중요한 구현 차이는 다음과 같다.

  • Original specification: value가 음수이면 그 값이 waiting thread 수를 반영할 수 있다.
  • Linux implementation: value가 0보다 작아지지 않을 수 있다.

이 차이는 구현 방식의 차이이지, 프로그래머가 기대하는 핵심 동작, 즉 token이 없으면 wait하고 post가 있으면 깨운다는 추상화는 동일하다.

18. Dining Philosophers와 Deadlock 예고

Dining Philosophers 예고는 Dining Philosophers를 언급한다. 이 문제는 semaphore와 deadlock의 고전적인 예제이다.

철학자들이 원형 테이블에 앉아 있고, 각 철학자는 양쪽 포크를 모두 가져야 식사할 수 있다고 하자. 모든 철학자가 동시에 왼쪽 포크를 잡고 오른쪽 포크를 기다리면 아무도 식사할 수 없다. 이것이 deadlock이다.

이 예제는 다음 노트의 deadlock 네 조건과 연결된다.

  • Mutual exclusion: 포크는 한 번에 한 철학자만 사용할 수 있다.
  • Hold and wait: 철학자는 한 포크를 잡은 채 다른 포크를 기다린다.
  • No preemption: 다른 철학자의 포크를 강제로 빼앗을 수 없다.
  • Circular wait: 각 철학자가 옆 철학자가 가진 포크를 기다리는 원형 대기가 생긴다.

19. 세 가지 Semaphore 사용 패턴 비교

1) Lock 패턴

  • 초기값: 1
  • 목적: critical section 보호
  • 예: mutex, binary semaphore
  • 코드 형태: wait → critical section → post

2) Ordering 패턴

  • 초기값: 0
  • 목적: 특정 event 이후에만 진행
  • 예: parent가 child 종료를 기다림
  • 코드 형태: waiter는 wait, event를 만든 thread는 post

3) Counting Resource 패턴

  • 초기값: resource 개수
  • 목적: 여러 개의 동일 자원 관리
  • 예: bounded buffer의 empty = MAX, full = 0
  • 코드 형태: resource 사용 전 wait, 반환/생성 후 post

구분 기준

Semaphore를 볼 때는 이름보다 초기값과 wait/post 위치를 먼저 봐야 한다. 초기값 1이면 lock일 가능성이 크고, 0이면 ordering/event일 가능성이 크고, MAX 같은 값이면 counting resource일 가능성이 크다.

20. 시험 대비 핵심 답안 프레임

Q. Semaphore란 무엇인가?

Semaphore는 정수 value와 대기 큐를 이용하는 동기화 primitive이다. sem_wait()는 value를 감소시키며 value가 충분하지 않으면 thread를 block하고, sem_post()는 value를 증가시키며 기다리는 thread를 깨울 수 있다. 초기값과 사용 위치에 따라 lock, ordering, counting resource로 사용할 수 있다.

Q. Producer/Consumer에서 왜 empty, full, mutex가 모두 필요한가?

empty는 빈 slot 수를 세어 producer가 buffer overflow를 일으키지 않게 하고, full은 채워진 item 수를 세어 consumer가 empty buffer에서 읽지 않게 한다. 그러나 이 둘은 fill, use, buffer 조작의 race condition을 막지 못하므로, 실제 put/get critical section은 mutex로 보호해야 한다.

Q. 왜 잘못된 Producer/Consumer 코드는 deadlock이 생기는가?

mutex를 먼저 잡은 뒤 full 또는 empty를 기다리면, 조건을 만족시켜야 하는 반대쪽 thread가 buffer를 조작하기 위해 필요한 mutex를 얻지 못한다. 따라서 기다리는 thread는 조건 변화를 기다리고, 반대쪽 thread는 mutex를 기다리는 순환 대기가 생긴다.

Q. Reader-Writer Lock의 장점과 단점은?

읽기 작업이 많은 workload에서 여러 reader가 동시에 들어갈 수 있어 병렬성이 증가한다. 하지만 writer와 reader 사이의 공정성을 잘 설계하지 않으면 reader가 계속 들어오는 동안 writer가 오래 기다리는 starvation이 발생할 수 있다.

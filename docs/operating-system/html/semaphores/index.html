1. 핵심 개념

3. Binary Semaphore: Lock으로 쓰는 경우

세마포어를 상호 배제(Mutex) 역할을 하는 Lock처럼 사용하는 패턴입니다.

sem_t m;
sem_init(&m, 0, 1); // 초기값을 1로 설정!

sem_wait(&m);
// critical section here
sem_post(&m);

초기값이 1이라는 것은 “critical section에 들어갈 수 있는 허가증(token)이 처음에 딱 하나 존재한다”는 뜻입니다.

실행 흐름 시뮬레이션

  1. 처음 value는 1입니다.
  2. Thread A가 sem_wait(&m)을 호출합니다. value가 1이므로 A는 이를 0으로 만들고 즉시 통과합니다.
  3. Thread A가 critical section을 실행하는 동안 Thread B가 sem_wait(&m)을 호출하면 value가 0이므로 대기 상태로 들어갑니다.
  4. A가 sem_post(&m)을 호출하여 value를 1로 돌려주고 대기 중이던 B를 깨우면, B가 비로소 들어가 실행합니다.
⚠️ 시험 함정 주의: Lock과 완전히 같을까?

용도는 비슷하지만 완전히 같지는 않습니다. 일반적인 mutex lock은 "락을 잡은(acquire) 스레드만 락을 풀(release) 수 있다"는 ownership 개념이 엄격하게 지배합니다. 반면 semaphore는 A 스레드가 wait하고 B 스레드가 post하는 교차 호출이 문법적으로나 원리적으로나 자연스럽게 가능합니다. 이 유연성 덕분에 lock뿐만 아니라 순서 제어(ordering)에도 유용하게 쓰입니다.

2. 수식과 정의

2. POSIX Semaphore API의 의미

POSIX semaphore의 기본 API와 작동 메커니즘을 분석합니다.

int sem_init(sem_t *s, int pshared, unsigned int value);
int sem_wait(sem_t *s);
int sem_post(sem_t *s);

1) sem_init()

sem_init(&s, 0, value)는 semaphore s를 초기값 value로 초기화합니다.

  • pshared = 0: 같은 process 내부의 thread들이 semaphore를 공유합니다. (대부분의 실습 예제)
  • pshared != 0: 서로 다른 process들이 semaphore를 공유합니다. 이 경우 semaphore 객체는 shared memory 영역에 있어야 합니다.

2) sem_wait()

sem_wait()는 전통적으로 P 연산 또는 down 연산이라고 부릅니다. 직관적으로는 다음과 같이 동작합니다.

  1. semaphore value가 양수이면 value를 1 감소시키고 즉시 통과합니다.
  2. value가 0이면 통과할 token이 없으므로 sleep/block 상태로 들어갑니다.
  3. 나중에 다른 thread가 sem_post()를 호출하면 깨어나 다시 진행합니다.

3) sem_post()

sem_post()는 V 연산 또는 up 연산이라고 부릅니다.

  1. semaphore value를 1 증가시킵니다.
  2. 기다리는 thread가 있으면 그중 하나를 깨웁니다.
📌 "Negative Value"에 관한 진실

개념적(고전적) 명세에서는 sem_wait()가 값을 무조건 감소시키므로 대기자가 있을 때 value가 음수가 될 수 있으며, 그 절댓값이 대기 큐의 크기를 반영한다고 설명합니다. 그러나 실제 Linux 구현에서는 value가 0 미만으로 내려가지 않으며 대기 스레드는 별도의 wait queue에서 관리됩니다. 시험 서술 시에는 이 두 메커니즘을 모두 언급하는 것이 안전합니다.

3. 직관과 그림

0. 동기화 프리미티브의 큰 그림

이 노트의 핵심은 semaphore가 단순히 lock의 다른 이름이 아니라, 정수값을 이용해 여러 종류의 동기화 문제를 표현하는 일반적인 primitive라는 점이다. 이전 노트에서 다룬 lock과 condition variable은 각각 역할이 분명했습니다.

  • Lock: critical section에 동시에 하나의 thread만 들어가게 한다.
  • Condition variable: 어떤 조건이 참이 될 때까지 thread를 잠재웠다가, 조건을 바꾼 thread가 깨운다.

Semaphore는 이 둘을 모두 흉내낼 수 있습니다. 단, 이것이 장점이면서 동시에 위험한 점입니다. semaphore 하나로 여러 의미를 표현할 수 있기 때문에, 코드만 보고도 “이 semaphore는 lock 역할인가, event ordering 역할인가, resource count 역할인가?”를 구분해야 합니다.

🧠 핵심 사고법

Semaphore의 value는 “지금 당장 통과 가능한 허가증(token)의 개수”라고 생각하면 좋다. sem_wait()는 허가증 하나를 가져가는 연산이고, 허가증이 없으면 기다린다. sem_post()는 허가증 하나를 돌려주는 연산이며, 기다리는 thread가 있으면 하나를 깨울 수 있다.

4. 예제

4. Semaphore로 Ordering(순서) 표현하기

세마포어를 활용해 condition variable처럼 특정 순서를 강제하는 정밀 제어 기법입니다.

sem_t s;

void *child(void *arg) {
    printf("child\n");
    sem_post(&s);
    return NULL;
}

int main(int argc, char *argv[]) {
    pthread_t c;
    sem_init(&s, 0, X); // 과연 X 값은 무엇이어야 할까?
    printf("parent: begin\n");
    pthread_create(&c, NULL, child, NULL);
    sem_wait(&s);
    printf("parent: end\n");
    return 0;
}

우리의 목표는 항상 다음과 같은 출력 순서를 보장하는 것입니다.

parent: begin
child
parent: end

왜 X는 반드시 0이어야만 하는가?

sem_init(&s, 0, 0)으로 초기화하면 처음에는 쓸 수 있는 토큰이 0개입니다. 따라서 부모 스레드가 sem_wait(&s)를 호출하면 토큰이 없으므로 무조건 대기하게 됩니다. 오직 자식 스레드가 실행을 완료하고 sem_post(&s)를 해주어야 토큰이 1이 되면서 부모가 대기를 깨고 parent: end를 출력하게 됩니다.

만약 X = 1로 설정했다면 부모는 대기 없이 곧바로 sem_wait(&s)를 통과하게 되므로 자식의 실행 여부와 관계없이 parent: end가 먼저 출력될 수 있는 치명적인 경쟁 조건이 발생합니다.

7. 완벽한 세마포어 해법

교착 상태를 피하는 정확한 해법은 매우 심플합니다. **상태 조건을 확인하는 대기 연산(`empty`, `full`)을 락 획득(`mutex`)보다 먼저 수행하도록 하는 것입니다.**

✅ 안전한 올바른 해법 코드

// [PRODUCER] 
void *producer(void *arg) {
    int i;
    for (i = 0; i < loops; i++) {
        sem_wait(&empty); // 1. 공간 확보 (대기 시 락 쥐지 않음!)
        sem_wait(&mutex); // 2. 그 다음 락 잡기
        put(i);
        sem_post(&mutex); // 3. 락 해제
        sem_post(&full);  // 4. 아이템 적재 알림
    }
}
// [CONSUMER] 
void *consumer(void *arg) {
    int i;
    for (i = 0; i < loops; i++) {
        sem_wait(&full);  // 1. 아이템 확보
        sem_wait(&mutex); // 2. 그 다음 락 잡기
        int tmp = get();
        sem_post(&mutex); // 3. 락 해제
        sem_post(&empty); // 4. 빈 공간 발생 알림
    }
}
이 구조에서는 생산자나 소비자가 공간/아이템 부족으로 대기 큐에 잠들 때 Mutex 락을 소유하지 않은 상태이므로, 다른 편 스레드가 자유롭게 락을 획득하여 자료구조를 안전하게 조작하고 대기를 상호 깨워줄 수 있습니다.

9. 세마포어의 자체 구현 (락 & CV 활용)

기존의 동기화 프리미티브인 **Mutex Lock과 Condition Variable**만을 이용하여 하위 계층에서 세마포어(Sem_t)의 완전한 추상화를 구현하는 수학적/논리적 모델입니다.

typedef struct __Sem_t {
    int value;            // 가용한 토큰 수
    pthread_cond_t cond;  // 대기 스레드를 잠재울 CV
    pthread_mutex_t lock; // 변수 보호용 뮤텍스
} Sem_t;

void Sem_init(Sem_t *s, int value) {
    s->value = value;
    pthread_cond_init(&s->cond, NULL);
    pthread_mutex_init(&s->lock, NULL);
}

void Sem_wait(Sem_t *s) {
    pthread_mutex_lock(&s->lock);
    while (s->value <= 0) // 토큰이 없으면
        pthread_cond_wait(&s->cond, &s->lock); // 대기 잠듬
    s->value--;
    pthread_mutex_unlock(&s->lock);
}

void Sem_post(Sem_t *s) {
    pthread_mutex_lock(&s->lock);
    s->value++;
    pthread_cond_signal(&s->cond); // 대기 중인 녀석 깨움
    pthread_mutex_unlock(&s->lock);
}
여기서도 왜 while 조건 대기 루프인가?

pthread_cond_wait에서 깨어났더라도 다른 제3의 스레드가 락을 먼저 꿰차서 토큰을 가져갔을 수 있는 경쟁 상태(Mesa Semantics)가 존재하므로, 반드시 while 문으로 감싸 토큰 소지율을 다시 점검한 뒤 차감해야 안전합니다.

5. 자주 틀리는 부분

5. Producer/Consumer 문제

생산자는 빈 공간에 아이템을 집어넣고, 소비자는 채워진 아이템을 꺼내는 Bounded Buffer 환경에서는 다음의 두 가지 동기화 조건이 만족되어야 합니다.

  1. Condition Synchronization: 생산자는 버퍼가 꽉 찼으면 빈 자리가 생길 때까지 대기해야 하고, 소비자는 버퍼가 비었으면 아이템이 들어올 때까지 대기해야 한다.
  2. Mutual Exclusion: 원형 버퍼(Circular Buffer)의 fill, use 변수 및 실제 배열 공간은 동시에 여러 스레드가 건드리면 인덱스 꼬임 등의 race condition이 발생하므로 상호 배제가 필수적이다.

원형 버퍼 구조식

int buffer[MAX]; // bounded buffer
int fill = 0;
int use = 0;

void put(int value) {
    buffer[fill] = value;
    fill = (fill + 1) % MAX;
}

int get() {
    int tmp = buffer[use];
    use = (use + 1) % MAX;
    return tmp;
}

empty / full 세마포어 정의

sem_t empty, full;
sem_init(&empty, 0, MAX); // 처음에 MAX개의 슬롯이 비어 있음
sem_init(&full, 0, 0);    // 처음에 0개의 아이템이 채워져 있음
  • empty: 버퍼의 빈 슬롯 수를 나타내는 상태 조건 세마포어. 생산자가 작업 전 wait하고 소비자가 작업 후 post한다.
  • full: 버퍼 내 유효한 아이템 수를 나타내는 상태 조건 세마포어. 소비자가 작업 전 wait하고 생산자가 작업 후 post한다.
스레드 락 결합 주의: 단일 생산자와 단일 소비자 환경에서는 인덱스 변경자(`fill`과 `use`)가 스레드별로 격리되어 경쟁하지 않으므로 상호 배제 락이 없어도 우연히 안전하게 작동할 수 있습니다. 그러나 멀티 생산자 / 멀티 소비자 환경으로 넘어가면 반드시 Mutex 락을 통한 상호 배제 처리가 결합되어야 합니다.

6. 잘못된 Mutex와 데드락

상호 배제를 달성하기 위해 단순히 Mutex를 씌우는 시도를 할 때, Mutex 락 획득 연산이 조건 대기 연산보다 바깥쪽에 위치하게 되면 파멸적인 데드락이 발생합니다.

🚨 치명적인 오류 코드 예시

// [PRODUCER] 
sem_wait(&mutex); // 락 획득 먼저!
sem_wait(&empty); // 빈 버퍼 대기
put(i);
sem_post(&full);
sem_post(&mutex);
// [CONSUMER] 
sem_wait(&mutex); // 락 획득 먼저!
sem_wait(&full);  // 가득 찬 버퍼 대기
tmp = get();
sem_post(&empty);
sem_post(&mutex);

데드락의 발생 시나리오 (버퍼가 완전히 비어있는 상황)

  1. 소비자가 먼저 실행되어 sem_wait(&mutex)를 성공적으로 마치고 락을 독점합니다.
  2. 소비자가 sem_wait(&full)을 호출합니다. 현재 버퍼는 비어 있으므로(full=0) 소비자는 아이템이 생산되기를 기다리며 잠에 빠집니다. **이때 소비자는 Mutex 락을 소유한 채 잠에 듭니다.**
  3. 생산자가 깨어나 아이템을 넣기 위해 sem_wait(&mutex)를 호출합니다. 하지만 락은 소비자가 잠든 채 쥐고 있으므로 생산자는 락을 얻지 못해 대기 큐로 들어갑니다.
  4. 소비자는 생산자가 아이템을 넣어주기(`full`을 post하기)를 기다리고, 생산자는 소비자가 락을 놔주기(`mutex`를 post하기)를 기다리는 **순환 대기(Circular Wait)가 완성되어 영원히 교착됩니다.**

8. Reader-Writer Lock의 설계와 한계

읽기 연산(Read)이 지배적으로 많고 쓰기 연산(Write)이 매우 드문 자료구조에서는 모든 스레드의 동시 진입을 막는 일반 Mutex가 극심한 성능 낭비가 될 수 있습니다. **"동시 다중 읽기는 허용하되, 쓰기는 오직 독점적으로 단 한 명만 수행하게 하는 락"**이 바로 Reader-Writer Lock입니다.

Reader-Writer Lock 구조체 설계

typedef struct _rwlock_t {
    sem_t lock;      // readers 변수를 보호하기 위한 Mutex
    sem_t writelock; // writer 및 첫 reader가 획득하는 큰 문 락
    int readers;     // 현재 자료구조를 읽고 있는 reader 수
} rwlock_t;

void rwlock_init(rwlock_t *rw) {
    rw->readers = 0;
    sem_init(&rw->lock, 0, 1);
    sem_init(&rw->writelock, 0, 1);
}

구현 함수 상세 분석

Writer 전용 락 함수 
void rwlock_acquire_writelock(rwlock_t *rw) {
    sem_wait(&rw->writelock);
}

void rwlock_release_writelock(rwlock_t *rw) {
    sem_post(&rw->writelock);
}

Writer는 독점을 확보해야 하므로 무조건 writelock 문을 걸어 잠그고 들어옵니다.

Reader 전용 락 함수 
void rwlock_acquire_readlock(rwlock_t *rw) {
    sem_wait(&rw->lock);
    rw->readers++;
    if (rw->readers == 1) // 첫 번째 독자!
        sem_wait(&rw->writelock); // 작가 접근 완전 봉쇄
    sem_post(&rw->lock);
}

void rwlock_release_readlock(rwlock_t *rw) {
    sem_wait(&rw->lock);
    rw->readers--;
    if (rw->readers == 0) // 마지막 독자!
        sem_post(&rw->writelock); // 작가 진입 허용
    sem_post(&rw->lock);
}

두 번째 이후 독자들은 writelock을 거치지 않으므로 동시 병렬 읽기가 수행됩니다.

⚠️ 치명적인 설계 결함: Writer Starvation (기아 상태)

위의 고전적인 구현식은 Reader 편향형(Reader-preference)입니다. 만약 읽기 요청이 끊임없이 연속해서 계속 들어온다면, `readers` 카운트가 결코 0이 되지 않으므로 **대기 중인 Writer 스레드는 영원히 락을 얻지 못하고 굶주리는 Starvation에 빠집니다.** 이를 해결하기 위해 Writer가 대기 큐에 들어오면 이후의 신규 Reader를 대기시키는 등 공정성(Fairness)을 개선하는 심화 설계가 필수적입니다.

7. 시험/복습용 핵심 정리

1. CV 복습과 Join 패턴

Condition variable을 이용해 parent thread가 child thread의 종료를 기다리는 패턴을 다시 상기해 봅시다.

void thr_exit() {
    pthread_mutex_lock(&m);
    done = 1;
    pthread_cond_signal(&c);
    pthread_mutex_unlock(&m);
}

void thr_join() {
    pthread_mutex_lock(&m);
    while (done == 0)
        pthread_cond_wait(&c, &m);
    pthread_mutex_unlock(&m);
}

여기서 핵심은 done이라는 공유 변수가 condition이고, mutex m이 그 condition을 보호한다는 점입니다.

왜 while (done == 0)인가?

pthread_cond_wait()에서 깨어났다고 해서 곧바로 조건이 참이라고 보장할 수 없습니다. 이유는 다음과 같습니다.

  1. Spurious wakeup: 실제 signal이 없었는데도 깨어날 수 있습니다.
  2. 경쟁 상황: signal을 받은 뒤 lock을 다시 얻기 전에 다른 thread가 상태를 바꿀 수 있습니다.
  3. 여러 대기자: signal/broadcast 후 깨어난 thread 중 일부만 조건을 만족할 수 있습니다.

따라서 condition variable 패턴에서는 항상 if보다 while을 사용해 조건을 다시 확인하는 것이 안전합니다.

🔗 세마포어와의 연결: 이 join 패턴을 semaphore로 훨씬 간단하게 표현할 수 있음을 보게 된다. 예를 들어 child가 끝나기 전까지 parent를 기다리게 하고 싶다면, 초기값 0인 semaphore를 두고 parent는 sem_wait(), child는 sem_post()를 호출하면 된다.

10. 식사하는 철학자 예고 및 핵심 시험 요약

1. 식사하는 철학자(Dining Philosophers)와 데드락 4조건

5명의 철학자가 원탁에 앉아 양쪽의 포크를 모두 집어야만 식사할 수 있을 때, 모든 철학자가 동시에 자신의 왼쪽 포크만 집는다면 영원히 오른쪽 포크가 오기를 기다리며 전원 굶어 죽는 데드락에 도달합니다. 이 예제는 다음 장에서 다룰 **데드락의 필수 조건 4가지**를 극명하게 보여줍니다.

1. Mutual Exclusion (상호 배제)
포크는 동시 공유 불가.
2. Hold and Wait (점유 및 대기)
하나의 포크를 쥔 상태로 다른 포크 대기.
3. No Preemption (비선점)
타인이 쥔 포크를 강제로 뺏을 수 없음.
4. Circular Wait (순환 대기)
환형 사슬 모양의 꼬리잡기 대기 유발.

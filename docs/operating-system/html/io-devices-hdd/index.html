💡 핵심 고난도 면접 대비 Q&A

Q1. 하드웨어 드라이브가 물리 스케줄링을 자체적으로 처리할 수 있는데, 왜 OS 레벨에서 디스크 스케줄러가 또 필요한가요?

최근의 현대적인 물리 드라이브(HDD/SSD)는 복수 개의 요청을 동시에 받아두고 내부 컨트롤러 펌웨어단(NCQ 등)에서 정밀한 내부 스케줄링을 직접 수행할 수 있습니다. 하지만 여전히 OS 레벨 스케줄러가 필요한 이유는 '보안과 공정성' 및 '우선순위 관리' 때문입니다. OS는 특정 고신뢰 작업(예: 데이터베이스 로그 쓰기, 커널 가상 메모리 스왑)을 일반 사용자 백그라운드 파일 읽기보다 훨씬 긴급하게 처리해야 합니다. 드라이브 하드웨어는 이런 상위 계층의 문맥(Context)을 모릅니다. 따라서 OS 레벨에서 1차적인 스레드 우선순위에 입각하여 요청의 완급을 조율해 보내야 합니다.

Q2. 최신 저장 장치인 SSD(Solid State Drive)에서도 이 디스크 스케줄링 알고리즘들이 그대로 유용하게 동작하나요?

매우 중요한 날카로운 지적입니다! 유용하지 않습니다. SSD는 기계식 회전 물리 암과 바늘 헤드가 없는 순수 반도체 셀 플래시 메모리 기반 장치입니다. 물리 암이 목표 트랙까지 물리적으로 왕복 이동할 필요가 없기 때문에 탐색 시간($T_{seek}$)과 회전 지연 시간($T_{rotation}$)이 아예 0입니다! 따라서 SSD에 무리하게 SCAN이나 SSTF 스케줄러를 적용하는 것은 불필요한 계산 오버헤드만 늘리는 비효율적 행위입니다. 그래서 Linux 등 현대 OS는 SSD 같은 장치에는 복잡한 정렬 연산을 생략하고 요청 순서대로 곧바로 쏘아 보내는 단순 무스케줄러인 NOOP (No-Operation / None) 또는 단순 FIFO 방식을 적용해 처리합니다.

3. 하드디스크(HDD) 내부 구조와 입출력 소요 시간

💡 개념 정의

"하드디스크(HDD)는 동심원을 따라 회전하는 물리적인 원판(Platter)과 움직이는 디스크 암(Arm)을 이용해 데이터를 기계식으로 쓰고 읽는 장치이다."

HDD의 대표적인 물리 용어 알아보기

  • 원판 (Platter): 자성을 띈 미디어가 입혀진 얇은 원판입니다. 데이터를 실제로 영구히 보관하는 공간입니다. 앞/뒤 양면을 모두 데이터 보관 공간으로 사용합니다.
  • 트랙 (Track): 원판 위에 자성 물질로 조각된 일련의 동심원을 말합니다.
  • 섹터 (Sector): 각 트랙을 피자 조각처럼 나눈 최소 데이터 단위(주로 512바이트 또는 4KB)입니다.
  • 실린더 (Cylinder): 여러 장의 플래터가 같은 중심축으로 겹쳐 있을 때 동일 트랙 반지름을 가진 수직 집합입니다.
  • 디스크 헤드와 암 (Head & Arm): 원판 표면 위를 나노미터 크기 격차로 미끄러지며 자성 정보를 읽고 쓰는 극미세 바늘입니다.

💡 입출력 시간 계산법 (I/O Time Formulation)

디스크에서 하나의 블록을 완전히 읽는 데 걸리는 전반적인 총 시간은 물리 부품들의 물리적 작동 움직임 시간의 합입니다:

$$T_{I/O} = T_{seek} + T_{rotation} + T_{transfer}$$
  • $T_{seek}$ (탐색 시간): 디스크 헤드가 있는 암이 데이터가 있는 올바른 트랙(반경)까지 가는 데 걸리는 시간. (기계적 이동이 가장 오래 걸림!)
  • $T_{rotation}$ (회전 지연 시간): 원판이 빙글빙글 돌아 요청한 섹터가 헤드 바로 밑 위치에 안착할 때까지 기다리는 평균 대기 시간.
  • $T_{transfer}$ (전송 시간): 원하는 섹터가 헤드 밑을 통과하면서 실제로 데이터를 자성 변조로 읽거나 쓰는 순수 전기 신호 변환 시간.

직관적 비유: 타겟 과녁 레코드 플레이어

디스크는 일종의 고속 회전하는 축음기 판입니다. 원하는 트랙을 찾기 위해 바늘 암을 들고 수평으로 조심스럽게 타겟 원 지점으로 위치시키는 것($T_{seek}$)이 총 동작 시간의 대부분을 점유합니다. 그 후 판이 휙휙 돌면서 정확한 재생 시작 지점이 바늘 위치 밑을 딱 스치고 지나가는 타이밍($T_{rotation}$)을 한 박자 쉬면서 기다리게 됩니다.

3. 직관과 그림

1. 시스템 아키텍처와 계층적 버스 구조

💡 개념 정의

"컴퓨터 내부 장치들을 서로 연결하는 통로인 '버스(Bus)'는 속도와 비용의 균형을 맞추기 위해 계층형 구조로 설계된다."

왜 버스를 속도별로 분리해서 설계해야 할까요?

모든 하드웨어 장치를 단 하나의 전선(버스)에 연결하면 편하겠지만, 성능상 치명적인 문제가 발생합니다. CPU와 메인 메모리 같은 초고속 장치가 키보드나 HDD 같은 저속 장치 때문에 대기하게 되는 병목 현상이 발생하기 때문입니다. 또한 고성능 물리 버스는 설계 비용이 극도로 비쌉니다.

이를 해결하기 위해 운영체제와 하드웨어 아키텍처는 버스를 계층화하여 물리적으로 나눕니다:

  • 메모리 버스 (Memory Bus): CPU와 메인 메모리를 다이렉트로 연결하는 가장 빠른 고성능 버스입니다.
  • 일반 I/O 버스 (General I/O Bus): 그래픽 카드 등 상대적으로 고속인 I/O 장치를 연결합니다. 대표적으로 PCIe (PCI)가 있습니다.
  • 주변장치 I/O 버스 (Peripheral I/O Bus): HDD, 키보드, 마우스 등 느린 장치들을 연결하는 범용 버스입니다. SATA, SCSI, USB 등이 여기에 속합니다.
직관적 비유: 도시의 도로망

메모리 버스는 '시속 150km 무제한 아우토반 고속도로'입니다. 고속으로 달릴 수 있는 스포츠카(CPU, RAM) 전용 도로죠. 반면 주변장치 I/O 버스는 동네 골목길이나 국도입니다. 만약 경운기(마우스, 키보드)를 아우토반에 진입시킨다면 도로 전체의 속도가 경운기 속도로 떨어지겠죠? 그래서 고속도로와 국도를 연결해 주는 나들목(Bridge/Controller)을 두어 구분하는 것입니다.

Q. CPU와 장치 인터페이스는 물리적으로 어떻게 통신하나요?

모든 I/O 장치 내부에는 OS가 장치를 제어할 수 있도록 인터페이스(Registers)를 가지고 있습니다. 주로 3가지 레지스터가 쓰입니다:

1. 상태 레지스터 (Status)
장치가 현재 바쁜지(BUSY), 데이터를 읽을 준비가 되었는지(READY) 상태를 나타냅니다.
2. 명령 레지스터 (Command)
CPU가 장치에게 내릴 명령(예: READ, WRITE)을 쓰는 곳입니다.
3. 데이터 레지스터 (Data)
실제 주고받을 데이터를 담아두는 버퍼 역할을 하는 공간입니다.

2. I/O 프로토콜: Polling vs Interrupt vs DMA

💡 개념 정의

"I/O 장치에 작업을 요청하고 데이터를 전송받는 과정에서 발생하는 CPU의 낭비를 막기 위한 통신 흐름 제어 방식들이다."

동작 메커니즘의 차이

초창기 방식인 폴링(Polling)과 PIO(Programmed I/O)는 데이터 전송의 매 순간마다 CPU가 개입하여 장치의 상태 레지스터를 끊임없이 확인하고, 메모리와 장치 버퍼 간에 데이터를 한 땀 한 땀 직접 옮겼습니다. 이로 인해 CPU가 쓸데없는 대기 시간에 과도하게 소모되는 문제가 생겼습니다.

1. Polling (폴링)
장치가 준비될 때까지 CPU가 무한 루프를 돌며 상태 레지스터만 체크하는 단순하지만 매우 비효율적인 방식.
2. Interrupt (인터럽트)
요청 후 CPU는 다른 프로세스를 처리하고, 장치가 준비되면 하드웨어 신호(Interrupt)를 보내 CPU가 작업을 이어서 받도록 하는 비동기 방식.
3. DMA (직접 메모리 접근)
대용량 데이터를 보낼 때, CPU 개입 없이 메모리와 I/O 장치가 직접 데이터를 주고받을 수 있도록 독립된 하드웨어 엔진(DMA controller)을 사용하는 기술.
직관적 비유: 피자 주문과 택배 수령

- 폴링(Polling): 피자를 주문해 두고, 인터폰 앞에 서서 10초에 한 번씩 "피자 왔나요?"라고 계속 확인하며 아무 일도 못 하는 상태.
- 인터럽트(Interrupt): 피자를 주문해 두고 방에서 밀린 공부를 하다가 초인종이 울리면 문을 열어주는 방식.
- DMA: 100박스의 대형 이삿짐이 올 때, 내가 직접 받아서 정리하지 않고, 이삿짐 센터 직원(DMA Controller)에게 "안방에 전부 놓아주시고 완료되면 사인만 해주세요"하고 자리를 비우는 방식.

4. 디스크 스케줄링 알고리즘 (Disk Scheduling Visualizer)

💡 개념 정의

"하드웨어 기계적 움직임을 최소화하고 데이터 처리 처리량을 높이기 위해, 들어온 요청들을 재배치하여 효율적인 순서대로 디스크 트랙을 방문하게 제어하는 기법이다."

핵심 5대 알고리즘 분석

탐색 시간($T_{seek}$)을 극도로 줄이기 위해, 무조건 먼저 들어온 요청 순서대로 헤드를 가차 없이 왔다 갔다 하는 비효율을 고쳐야 합니다.

알고리즘 원리 및 장점 단점 / 한계
FCFS (First-Come, First-Served) 선착순 처리. 극도로 직관적이며 기아 상태(Starvation)가 일어나지 않음. 요청이 꼬여있을 때 헤드가 지그재그로 디스크 전체를 계속 왕복하여 탐색 성능이 매우 처참함.
SSTF (Shortest Seek Time First) 현재 헤드 위치에서 가장 가까운 트랙 거리를 지닌 요청부터 찾아가서 즉각 해결함. 기아 상태(Starvation) 유발 가능. 외곽 트랙의 요청들은 헤드가 가운데 트랙을 맴도는 과정에서 평생 굶을 수 있음.
SCAN (엘리베이터) 엘리베이터처럼 한 방향으로 끝까지 쭉 움직이면서 싹 쓸고 해결한 뒤, 반대로 틀어서 다시 복귀하며 해결함. 가장자리 트랙은 평균 대기 시간이 증가하는 불균형 발생. 또한 회전 방향 전환 지연 무시.
C-SCAN (Circular SCAN) 한 방향으로만 가며 스위핑한 뒤, 끝에 닿으면 데이터를 훑지 않고 시작 지점으로 빠르게 무부하 고속 복구 후 다시 한 방향으로만 일방 처리. 평균적인 응답 속도가 SCAN보다 더 고르고 안정적이지만 복귀 시 일시적 지연 발생.
SPTF (Shortest Positioning Time First) 트랙 거리만 고려하지 않고, 회전 시간 지연까지 복합 계산하여 가장 빠르게 도착할 수 있는 트랙과 섹터 선택. 운영체제가 하드웨어 드라이브의 미세한 기기 특성을 실시간 예측하기 매우 어렵고 구현이 무거움.

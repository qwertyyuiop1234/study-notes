Concurrency and Threads

A full study note on the OS mechanisms behind threads, memory layouts, scheduling nondeterminism, race conditions, critical sections, mutexes, and condition variables.

Threads Mutex Condition Variables OS Fundamentals

1. Thread definition and memory layout

A thread is an independent CPU execution flow inside one process, while threads in the same process share the address space such as heap and code.

Process versus thread

A traditional process runs independently with a fully isolated memory space. In a multithreaded program, several execution flows live inside one large shared address space. Each thread still needs its own execution state: a program counter (PC), a private set of registers, and its own stack for return addresses, parameters, and local variables. The heap, where shared global or dynamically allocated data can live, and the compiled code region are shared.

Thread context switch

A process context switch is expensive because the OS must switch to a different virtual-memory mapping such as another page table. A thread context switch inside the same process is much lighter: the OS saves the outgoing thread's PC and registers into its TCB (Thread Control Block), then restores the next thread's TCB state. The page table does not need to be replaced because the address space is the same.

Single-threaded address space

  • Program Code
  • Heap — shared resource area
  • One Stack

Multithreaded address space

  • Program Code — shared
  • Heap — shared
  • Stack 1 — Thread A
  • Stack 2 — Thread B

Intuition: roommates under one roof

A process is like a separate apartment household. Accessing another household's kitchen or tools requires heavier communication such as IPC or pipes. Threads are like roommates in one apartment: they freely share the living room and kitchen, meaning the shared address space, heap, and code, but each has a private bedroom and phone, meaning stack, registers, and PC.

Why not share the stack? If several threads shared one stack while running different functions, return addresses, arguments, and local variables would become tangled. Independent stacks are technically necessary for independent control flow.

TL;DR: A thread is a lightweight process-like execution flow. It can switch without changing the virtual-memory table; code and heap are shared, while PC, registers, and stack are private.

2. Why use threads?

Threads are used for computational parallelism on multicore hardware and for keeping a program useful while slow I/O would otherwise block execution.

1) Parallelism

Modern hardware cannot endlessly increase one CPU core's speed, so systems use multiple cores. Threads let a large computation be split into independent pieces and run on several cores at the same time, dramatically improving throughput when the work is parallelizable.

2) Avoiding blocking latency

Disk reads and network communication are extremely slow compared with CPU operations. In a single-threaded program, one blocking I/O call can stop the whole process until data arrives. In a multithreaded program, if one thread blocks on I/O, the OS scheduler can switch to another ready thread and keep the CPU doing useful work.

Intuition: supermarket checkout lines

With a single checkout line, one customer's refund or price check delays everyone behind them. With multiple lines, other checkout lanes continue serving customers, so the whole store remains productive.

TL;DR: Threads exploit multiple CPUs for faster computation and bypass waiting time by giving idle cores other ready work while a thread waits for I/O.

3. Thread creation and scheduler nondeterminism

Threads can be created with system APIs, but once they start, the exact execution order is controlled by the OS scheduler and is nondeterministic.

POSIX thread creation in C

On Linux/Unix systems, multithreaded C programs commonly use pthread. pthread_create() starts a new thread, and pthread_join() lets the parent wait until the thread finishes so resources can be reclaimed.

#include <pthread.h>
#include <stdio.h>
#include <assert.h>

void *mythread(void *arg) {
    printf("%s\n", (char *) arg);
    return NULL;
}

int main(int argc, char *argv[]) {
    pthread_t p1, p2;
    int rc;

    printf("main: begin\n");
    rc = pthread_create(&p1, NULL, mythread, "A"); assert(rc == 0);
    rc = pthread_create(&p2, NULL, mythread, "B"); assert(rc == 0);
    rc = pthread_join(p1, NULL); assert(rc == 0);
    rc = pthread_join(p2, NULL); assert(rc == 0);
    printf("main: end\n");
    return 0;
}

Nondeterministic scheduling

If this simple program is executed many times, the output order is not guaranteed to be identical. Sometimes A appears first, sometimes B appears first, and sometimes both worker threads finish before the main thread reaches the join calls. The fine-grained timing depends on the internal state and decisions of the OS scheduler. This is nondeterministic execution.

Scenario 1: Thread 1 priority

  1. main prints “main: begin”
  2. Thread 1 and Thread 2 are created
  3. main waits for Thread 1
  4. Thread 1 prints “A”
  5. main waits for Thread 2
  6. Thread 2 prints “B”
  7. main prints “main: end”

Scenario 2: main moves quickly

  1. main prints “main: begin”
  2. Thread 1 is created
  3. Thread 1 immediately prints “A”
  4. Thread 2 is created
  5. main waits for both threads
  6. Thread 2 later prints “B”
  7. main prints “main: end”
TL;DR: pthread_create launches threads and pthread_join synchronizes with them, but the scheduler controls the exact timing, so execution is inherently nondeterministic.

4. Shared data and race conditions

A race condition occurs when multiple execution flows write shared data concurrently and the final result depends on timing rather than logical correctness.

The shared counter mystery

Suppose two threads each increment a global counter ten million times. The expected final value is exactly twenty million, but real executions can produce smaller and different values.

static volatile int counter = 0;

void *mythread(void *arg) {
    for (int i = 0; i < 1e7; i++) {
        counter = counter + 1; // critical section!
    }
    return NULL;
}

// Possible output:
// main: done with both (counter = 20000000) // lucky
// main: done with both (counter = 19345221) // lost updates
// main: done with both (counter = 19221041) // different every run

Why one C line is not one hardware step

The high-level statement counter = counter + 1; is not performed as one indivisible action. At the hardware level, it becomes a sequence: load the memory value into a CPU register such as %eax, increment that register, then store the register value back into memory.

A destructive context-switch interleaving

Thread 1 reads memory value 50 into its register and locally computes 51. Before it stores the value back, the scheduler preempts it. Thread 2 also reads the still-unchanged memory value 50, computes 51, and stores 51. When Thread 1 resumes, it stores its old computed 51 as well. Two increments happened, but memory increased only from 50 to 51 instead of 52. One update was lost.

Why call this a race? Because the correctness of the final state depends on which thread reaches the memory write first, like a race whose winner changes the outcome.

5. Critical sections and mutual exclusion

Mutual exclusion is the principle that at most one thread may enter a critical section at a time.

Critical section

A critical section is the part of code that can cause a race condition because it accesses and writes shared resources such as global variables or shared data structures. In the counter example, counter = counter + 1; is the critical section.

Mutual exclusion

Mutual exclusion prevents another thread from entering the same dangerous region while one thread is already executing it. It locks the region so the shared memory cannot be modified by multiple threads at the same time.

Intuition: airplane restroom lock

An airplane restroom is a critical section. Once a passenger enters and locks the door, everyone else must wait until the passenger finishes and unlocks it. The lock enforces mutual exclusion.

TL;DR: A critical section is code that can break when multiple threads touch it together. Mutual exclusion ensures only one thread enters at a time.

6. Atomicity and mutex practice

Atomicity means all-or-nothing execution: an operation completes as one indivisible unit or does not occur, with no other operation interleaving in the middle.

Limits of hardware atomic operations

Hardware may provide atomic machine instructions for simple integer operations, but those are limited. They cannot directly protect complex algorithms such as safely transforming a tree structure or sorting and updating a long list. Operating systems and libraries build general synchronization primitives on top of lower-level hardware support. A central primitive is the mutex.

Mutex API summary

pthread_mutex_lock()

Acquire the lock before entering the critical section. If another thread owns it, block until it is available.

pthread_mutex_unlock()

Release the lock after finishing so another waiting thread can proceed.

pthread_mutex_trylock()

Try to acquire the lock; if it is already held, return an error immediately instead of waiting.

pthread_mutex_timedlock()

Wait only up to a specified timeout; if the lock is not released in time, give up.

Safe counter increment with a mutex

#include <pthread.h>

pthread_mutex_t lock = PTHREAD_MUTEX_INITIALIZER;
// Or dynamically: pthread_mutex_init(&lock, NULL);

void *mythread(void *arg) {
    for (int i = 0; i < 1e7; i++) {
        pthread_mutex_lock(&lock);
        counter = counter + 1; // only one thread can execute this region
        pthread_mutex_unlock(&lock);
    }
    return NULL;
}

// When finished: pthread_mutex_destroy(&lock);
What if a thread locks but forgets to unlock? The program can fall into deadlock. The thread that took the key never returns it, so other threads wait forever and the system can become stuck.

7. Condition variables and signaling

A condition variable is an event-signaling mechanism that safely coordinates ordering between threads when one action must happen before another can proceed.

Wastefulness of spinning

If Thread B wants to wait until a shared flag ready == 1, the crude method is busy waiting:

while (ready == 0) {
    // spin / busy wait
}

This wastes CPU cycles, can keep useful threads from running, and can interact badly with weak memory consistency models or compiler optimization.

Elegant waiting with condition variables

A pthread_cond_t lets a thread sleep instead of burning CPU. pthread_cond_wait(cond, mutex) atomically releases the mutex and puts the thread to sleep; otherwise no other thread could change the condition. When the sleeping thread receives a signal, it wakes and reacquires the mutex before continuing. pthread_cond_signal(cond) wakes one waiting thread.

Canonical coding template

pthread_mutex_t lock = PTHREAD_MUTEX_INITIALIZER;
pthread_cond_t cond  = PTHREAD_COND_INITIALIZER;
int ready = 0;

void *waiter_thread(void *arg) {
    pthread_mutex_lock(&lock);
    while (ready == 0) { // use while, not if, to handle spurious wakeups
        pthread_cond_wait(&cond, &lock);
    }
    // safely perform the dependent work
    pthread_mutex_unlock(&lock);
    return NULL;
}

void *worker_thread(void *arg) {
    pthread_mutex_lock(&lock);
    ready = 1;
    pthread_cond_signal(&cond);
    pthread_mutex_unlock(&lock);
    return NULL;
}
Intuition: cafe pager Standing at the counter and asking “Is my drink ready?” every half-second is spinning. Taking a pager and sitting down is a condition variable: wait comfortably, then respond when the pager signals.

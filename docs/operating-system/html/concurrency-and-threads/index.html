0. 전체 지도: 이 노트에서 잡아야 할 것

동시성(concurrency)은 현대 운영체제와 고성능 애플리케이션의 근간입니다. 본 단원에서는 프로세스의 뒤를 잇는 경량 실행 흐름인 스레드를 이해하고, 왜 동시성 처리가 필연적으로 경쟁 상태(race condition)를 유발하는지, 그리고 뮤텍스(mutex)와 조건 변수(condition variable)가 왜 필요한지 총망라합니다.

먼저 이해할 것

  • 단일 프로세스의 메모리 구조 (Stack, Heap, Code)
  • CPU Context Switch의 전반적 메커니즘
  • 프로세스 스케줄러의 비결정성(Nondeterminism)

최종적으로 할 수 있어야 할 것

  • 스레드와 프로세스의 차이를 메모리 공유 유무로 서술
  • 임계 영역(Critical Section)의 발생 원인 증명
  • 뮤텍스 및 조건 변수의 올바른 바인딩 패턴 적용
읽는 법: 개념 → 메모리 구조 → 경쟁 상황 분석 → 뮤텍스/CV 해법 순서로 학습하며, 시뮬레이터를 통해 하드웨어 레벨의 경쟁을 직접 목격해보세요.

1. 핵심 개념

스레드 (Thread)은/는 프로세스 내의 독립적인 실행 단위이며, 같은 프로세스 내부의 스레드들은 Heap, Global Data, Code 영역을 자유롭게 공유합니다.

스레드 주소 공간

스레드는 독립된 Program Counter (PC), 레지스터 셋, 그리고 사설 스택(Private Stack)을 가집니다. 그 외 힙과 코드는 모든 스레드가 한 지붕 아래에서 공유합니다.

스레드 문맥 교환

프로세스 문맥 교환과 달리, 스레드 문맥 교환(Thread Context Switch)은 가상 메모리 테이블(Page Table)을 변경하지 않으므로 비용이 현격하게 저렴합니다. TCB(Thread Control Block)에 PC와 레지스터만 백업/복구하면 됩니다.

핵심 직관: 한 지붕 아래 룸메이트 프로세스가 완전히 독립된 개별 아파트 호수라면, 스레드는 한 아파트 아래 방을 나눠 쓰는 룸메이트들입니다. 거실과 부엌(힙, 코드)은 자유롭게 같이 쓰지만, 자기 방과 일기장(스택, PC/레지스터)은 따로 가집니다.

2. 수식과 정의

공유 데이터 증감에 락이 없을 때 발생하는 데이터 누락 공식과 임계 범위 논리입니다.

$$\text{counter} = \text{counter} + 1$$

한 줄의 고차 언어 코드는 어셈블리/하드웨어 레벨에서 다음과 같은 3개의 원자적이지 않은 과정으로 쪼개집니다.

기호/단계의미주의점
1. LOAD메모리의 값을 CPU 레지스터(%eax)로 로드이 단계 완료 후 선점(Preemption)당하면 값 유실 위험이 생김
2. INCR레지스터 내의 값을 1 증가레지스터 값만 바뀌며 메모리에는 아직 반영되지 않음
3. STORE레지스터의 값을 메모리 주소에 다시 기록동시에 여러 스레드가 쓰면 나중에 쓴 스레드가 앞선 값을 덮어씀

3. 직관과 그림

스레드가 주소 공간(Address Space)을 어떻게 공유하고 쪼개 쓰는지 보여주는 논리 서식입니다.

단일 스레드 프로세스
  • Program Code (공유)
  • Heap (공유)
  • Single Stack
멀티 스레드 프로세스
  • Program Code (공유)
  • Heap (공유)
  • Stack (Thread A 사설)
  • Stack (Thread B 사설)

4. 예제

POSIX thread API를 활용한 기본적인 C언어 스레드 생성과 동기화 예제입니다.

#include <pthread.h>
#include <stdio.h>
#include <assert.h>

void *mythread(void *arg) {
    printf("%s\n", (char *) arg);
    return NULL;
}

int main(int argc, char *argv[]) {
    pthread_t p1, p2;
    int rc;

    printf("main: begin\n");
    rc = pthread_create(&p1, NULL, mythread, "A"); assert(rc == 0);
    rc = pthread_create(&p2, NULL, mythread, "B"); assert(rc == 0);
    rc = pthread_join(p1, NULL); assert(rc == 0);
    rc = pthread_join(p2, NULL); assert(rc == 0);
    printf("main: end\n");
    return 0;
}

이 코드는 운영체제 스케줄러의 비결정성(Nondeterminism) 때문에 실행할 때마다 A와 B의 출력 순서가 달라질 수 있습니다.

5. 자주 틀리는 부분

동시성 동기화를 처리할 때 흔히 저지르는 오해와 교훈입니다.

오해 1

스레드는 가상 메모리 공간을 아예 공유하지 않는다? 아닙니다, stack을 제외한 code와 heap을 완전 공유합니다.

오해 2

counter = counter + 1;은 한 번에 일어나서 안전하다? 아닙니다, 어셈블리 단계에서 load-increment-store로 분할되므로 위험합니다.

오해 3

락을 걸기만 하면 모두 만사 해결이다? 락을 해제(unlock)하지 않거나 교차 획득하면 교착 상태(Deadlock)에 빠집니다.

해결 기준

임계 영역 내에서는 무조건 상호 배제(Mutual Exclusion)를 달성하기 위해 Mutex Lock을 적용해야 합니다.

시험 함정: 스레드가 락을 쥔 채 pthread_cond_wait가 아닌 일반 busy wait나 무한 루프에 돌입하면 시스템의 다른 스레드들이 락을 얻지 못하고 완전히 고갈되는 참사가 발생합니다.

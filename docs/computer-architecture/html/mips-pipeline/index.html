1. Single Cycle CPU의 한계

Single-cycle CPU는 하나의 명령어가 한 clock cycle 안에서 fetch, decode, execute, memory access, write-back을 모두 끝내도록 설계한다. 구조는 단순하지만 clock period가 가장 긴 instruction path에 맞춰진다.

대표적으로 lw는 instruction memory → register file → ALU address calculation → data memory → register write까지 지나가므로 critical path가 길다. 그 결과 단순한 R-format instruction이나 branch도 같은 긴 clock period를 강제로 사용한다.

핵심: single-cycle의 문제는 “모든 명령어를 똑같이 긴 한 cycle에 넣는다”는 점이다. Pipeline은 한 instruction의 전체 latency를 없애는 것이 아니라, 여러 instruction을 겹쳐 실행해서 throughput을 높인다.

Exam compression

  • Critical path: clock period를 결정하는 가장 긴 combinational path.
  • Single-cycle: control은 단순하지만 clock이 느려진다.
  • Pipeline: stage를 나누어 clock period를 줄이고 instruction throughput을 높인다.

2. Pipeline 법칙: Latency vs Throughput

빨래방 비유에서 한 빨래더미가 세탁→건조→개기를 모두 거치는 시간은 크게 줄지 않는다. 하지만 여러 빨래더미가 서로 다른 장비를 동시에 사용하면 일정 시간 후에는 더 자주 결과가 나온다.

  • Latency: 한 작업이 시작부터 끝까지 걸리는 시간.
  • Throughput: 단위 시간당 완료되는 작업 수.
  • Pipeline의 목표: latency보다 throughput 개선.

3. MIPS 5단계 Pipeline

IFInstruction Fetch. PC로 instruction memory를 읽고 다음 PC 후보를 준비한다.
IDInstruction Decode. Register file read, immediate sign extension, main control generation.
EXExecute. ALU operation, address calculation, branch comparison/target computation.
MEMMemory Access. Load/store가 data memory를 읽거나 쓴다.
WBWrite Back. ALU result 또는 memory data를 register file에 기록한다.

각 stage 사이에는 pipeline register가 있어 한 cycle의 결과를 다음 cycle의 입력으로 고정한다. 이 경계 덕분에 서로 다른 instruction들이 동시에 다른 stage에서 처리될 수 있다.

4. Pipeline 성능 분석

이상적인 $k$-stage pipeline에서 $n$개 instruction을 실행하는 cycle 수는 대략 다음과 같다.

$$\text{cycles} \approx k+n-1$$

처음 $k-1$ cycle은 pipeline을 채우는 시간이고, 이후에는 이상적으로 매 cycle마다 하나의 instruction이 완료된다. 실제 성능은 stage imbalance, pipeline register overhead, data/control hazard 때문에 이상치보다 낮아진다.

MIPS ISA가 pipeline에 유리한 이유

  • 고정 길이 instruction: fetch/decode가 규칙적이다.
  • Load/store architecture: memory access가 주로 MEM stage에 집중된다.
  • 규칙적인 register operand 위치: register read와 control generation이 단순해진다.
  • 단순 addressing mode: stage 균형을 맞추기 쉽다.

5. Pipeline Register의 역할

Pipeline register는 stage 사이의 격리벽이다. 각 stage는 cycle 동안 자신의 입력 register를 읽고 combinational logic을 수행한 뒤, cycle edge에서 다음 pipeline register에 결과를 저장한다.

IF/IDFetched instruction, PC+4 등 ID에 필요한 값 보존
ID/EXRead data, sign-extended immediate, register 번호, EX/MEM/WB control bundle
EX/MEMALU result, branch decision/target, store data, MEM/WB control
MEM/WBMemory read data 또는 ALU result, write register 번호, WB control

자주 하는 오해

Pipeline register는 단순히 “계산값 하나”만 넘기는 장치가 아니다. 뒤 stage에서 필요한 instruction field, destination register number, control bit까지 모두 instruction과 동기화되어 이동해야 한다.

6. Load 명령어의 Datapath 수정

lw rt, offset(rs)는 MEM stage에서 memory data를 얻고, WB stage에서 그 값을 rt에 쓴다. 문제는 destination register 번호인 rt가 ID stage에서 결정되지만 실제 사용은 WB stage에서 일어난다는 점이다.

따라서 rt 번호를 ID stage 이후에 버리면 WB stage는 어느 register에 써야 하는지 모른다. 해결책은 write register number를 data/control과 함께 ID/EX → EX/MEM → MEM/WB까지 계속 전달하는 것이다.

시험 답안: Load instruction의 destination register identifier는 write-back 시점까지 보존되어야 하므로 pipeline register의 일부로 전달되어야 한다.

Wrong vs Fixed

  • Wrong: WB stage에 data만 도착하고 destination id가 없음.
  • Fixed: data와 destination id가 같은 instruction context로 MEM/WB까지 이동.

7. Control Signal의 전파

Control signal은 보통 ID stage에서 opcode/funct field를 보고 한 번에 생성된다. 그러나 모든 control bit가 ID에서 즉시 쓰이는 것은 아니다. 어떤 것은 EX에서, 어떤 것은 MEM에서, 어떤 것은 WB에서 필요하다.

Control group대표 신호사용 stage의미
EXRegDst, ALUSrc, ALUOpEXALU 입력/동작과 destination 선택
MEMBranch, MemRead, MemWriteMEM분기 및 data memory 제어
WBRegWrite, MemtoRegWBregister write 여부와 write-back data source 선택

그래서 ID/EX register에는 EX/MEM/WB control bundle이 모두 들어가고, stage가 진행될수록 이미 사용된 control bit는 사라지고 필요한 bit만 계속 전달된다.

1. Single Cycle CPU의 한계와 파이프라인 개요

싱글 사이클 구현은 모든 명령어가 동일한 1클럭 주기를 가지므로, 가장 긴 경로(Critical Path)인 load 명령어에 의해 전체 클럭 주기가 강제 하향 평준화되는 비효율성을 갖는다.

싱글 사이클 CPU의 작동 한계점

싱글 사이클 하드웨어 설계는 아주 단순하고 이해하기 쉽다. 하나의 클럭 주기 동안 단 하나의 명령어만 완전히 시작되고 완료된다. 하지만 이 철학은 치명적인 성능 저하의 주범이 된다. CPU 클럭 주기의 길이는 가장 긴 지연을 갖는 명령어에 맞춰 결정되어야 하기 때문이다.

MIPS 명령어 중 가장 긴 작업 경로인 임계 경로(Critical Path)를 갖는 것은 대표적으로 lw이다. lw는 명령어 메모리 읽기 → 레지스터 파일 읽기 → ALU 주소 연산 → 데이터 메모리 읽기 → 레지스터 파일 다시 쓰기 단계를 모두 차례대로 통과해야 하므로 큰 전파 지연을 수반한다.

공통 케이스의 고속화 원칙 위배

컴퓨터 아키텍처의 황금 규칙인 “가장 흔하게 발생하는 일을 빠르게 만들라”는 원칙에 비추어 보면, 드물게 쓰이는 무거운 명령어 때문에 자주 실행되는 단순 연산의 속도까지 낮춰야 하는 싱글 사이클 구조는 비효율적이다. 명령어에 따라 클럭 주기를 매번 유동적으로 조절하는 것은 물리적 회로 설계 수준에서 현실적이지 않다.

직관적 비유: 대형 마트의 무식한 계산대

물건 하나를 산 사람과 카트 가득 물건을 채운 사람이 뒤섞인 계산대를 생각하자. 싱글 사이클은 카트 가득 채운 사람의 긴 계산이 완전히 끝날 때까지, 뒤에 있는 물건 하나만 가진 사람의 결제도 전혀 진행하지 않는 계산대와 같다.

TL;DR: 가장 긴 지연을 가지는 lw가 싱글 사이클의 최대 클럭 스피드를 제한한다. 해결책은 클럭 주기를 stage로 나누고 여러 명령어를 겹쳐 수행하는 pipelining이다.

2. 빨래방 비유로 이해하는 파이프라인 법칙

파이프라이닝은 개별 작업의 소요 시간(latency)을 본질적으로 단축시키지 않으며, 전체 작업 처리량(throughput)을 증가시키는 기술이다.

빨래방 공식

네 사람이 각각 한 더미씩의 빨래를 세탁, 건조, 개기 순서로 처리한다고 하자. 각 공정별 소요 시간은 세탁 30분, 건조 40분, 개기 20분이다.

순차적 빨래

Ann의 빨래가 세탁, 건조, 개기까지 모두 끝난 후에야 Brian이 세탁기에 빨래를 넣는다. 4개의 빨래 더미를 처리하려면 총 $90\text{분} \times 4 = 360\text{분}$이 걸린다.

파이프라인 빨래

Ann의 세탁이 끝나고 건조기로 넘어가자마자 Brian은 즉시 세탁기를 돌린다. 서로 다른 자원을 동시에 사용하므로 4더미의 처리가 약 210분에 마무리될 수 있다.

구분순차 처리파이프라인 처리
총 소요 시간6시간3.5시간
개별 latency90분90분, 거의 동일
throughput0.66 더미/시간1.14 더미/시간

아키텍처적 교훈

  • 개별 latency는 감소하지 않는다. Ann의 빨래는 여전히 끝날 때까지 전체 공정을 거친다.
  • 속도 향상비의 한계는 pipeline 단계 수와 관련된다. 이상적으로 $N$단계 pipeline은 최대 $N$배에 가까운 throughput 개선을 기대할 수 있다.
  • 가장 느린 단계가 전체 속도를 좌우한다. 건조기가 가장 느리면 그 단계가 병목이 된다. Pipeline 효율을 극대화하려면 단계 시간이 균형 잡혀야 한다.

TL;DR: Pipeline은 개별 명령어의 실행 시간보다 단위 시간당 끝나는 명령어 수를 증가시킨다. 가장 느린 공정이 전체 시스템 속도의 한계를 규정한다.

3. MIPS 5단계 파이프라인: IF-ID-EX-MEM-WB

MIPS 구조는 명령어 실행 과정을 독립된 5개의 물리적 stage로 쪼개어 동시 실행 흐름을 제어한다.

IF

PC가 가리키는 instruction memory에서 32비트 명령어를 읽고, PC+4를 준비한다.

ID

명령어를 decode하고 Rs/Rt register 값을 읽으며 immediate를 확장한다.

EX

ALU 연산, load/store 주소 계산, branch 비교와 target 계산을 수행한다.

MEM

lw, sw가 data memory를 읽거나 쓴다. 다른 명령어는 통과한다.

WB

ALU 결과나 load data를 register file의 목적지 register에 기록한다.

직관적 비유: 자동차 조립 공장

IF는 기본 뼈대 강판을 가져오고, ID는 엔진과 조립 설명서를 확인하며, EX는 실린더와 프레임을 용접하고, MEM은 내부 마감 공정을 처리하며, WB는 번호판을 붙이고 완성차로 등록하는 과정에 비유할 수 있다.

TL;DR: MIPS의 5대 핵심 공정은 IF → ID → EX → MEM → WB이며, 이 stage들이 컨베이어 벨트처럼 맞물려 전진한다.

4. 파이프라인 성능 분석과 MIPS ISA의 비밀

파이프라인의 1클럭 주기는 가장 느린 단일 stage의 지연 시간에 묶이지만, 균등하게 나뉜 하드웨어는 clock cycle time을 크게 줄일 수 있다.

실제 수치로 보는 차이

레지스터 읽기/쓰기 공정(ID, WB)이 각각 100ps, 나머지 공정(IF, EX, MEM)이 각각 200ps라고 하자. 싱글 사이클 구현은 가장 긴 lw 경로인 $200+100+200+200+100=800\text{ps}$가 한 cycle을 결정한다. 반면 pipeline 구현은 가장 느린 단일 공정인 200ps가 cycle time의 기준이 된다.

100개 명령어 처리 예시

Single-cycle

Cycle time = 45ns, CPI = 1
총 시간 = $45\text{ns} \times 100 = 4500\text{ns}$

Multicycle

Cycle time = 10ns, 평균 CPI = 3.6
총 시간 = $10\text{ns} \times 3.6 \times 100 = 3600\text{ns}$

Ideal pipeline

Cycle time = 10ns, CPI ≈ 1
총 시간 = $10\text{ns} \times (100+4)=1040\text{ns}$

따라서 성능 개선비는 $4500/1040 \approx 4.33$배가 된다. 실제로는 hazard와 pipeline register overhead 때문에 이상적인 수치보다 낮아질 수 있다.

MIPS ISA가 pipeline에 최적화된 이유

  • 모든 명령어가 32비트 고정 길이: IF와 ID가 매우 규칙적으로 진행된다.
  • 정형화된 명령어 형식: register 번호가 고정된 bit 위치에 있어 decode와 register read를 병렬적으로 준비하기 쉽다.
  • 엄격한 load/store architecture: memory 접근이 lw, sw로 제한되어 EX에서 주소를 계산하고 MEM에서 memory를 밟는 흐름이 규칙적이다.

TL;DR: Pipeline은 clock speed와 throughput을 개선한다. MIPS의 고정 길이 기계어와 단순한 register 배치가 pipeline 하드웨어 복잡도를 줄인다.

5. 파이프라인 레지스터와 정보 격리

Pipeline register는 각기 다른 cycle에서 처리 중인 서로 다른 명령어들이 서로의 연산 정보를 덮어쓰지 못하도록 stage 사이를 물리적으로 격리하는 checkpoint 장벽이다.

동시에 여러 명령어가 pipeline 안에서 흐를 때 stage 사이에 임시 저장 장치가 없다면, EX에서 ALU 연산을 수행하는 순간 뒤따라오던 ID stage의 값이 섞여 연산 장치를 망칠 수 있다. 따라서 각 stage 경계에는 매 clock마다 이전 결과를 잡아두고 다음 단계로만 밀어주는 pipeline register가 필요하다.

Pipeline register bit width 예시

IF/ID — 64 bits

instruction 32 bits + PC+4 32 bits.

ID/EX — 147 bits

control 9 bits, read data 1/2, sign-extended immediate, PC+4, Rt/Rd 번호 등.

EX/MEM — 107 bits

남은 control, branch target, ALU result, zero flag, store data, write register 번호.

MEM/WB — 71 bits

최종 control, memory read data, ALU result, write register 번호.

직관적 비유: 조립식 서랍 보관함

세탁이 끝난 옷을 다음 공정으로 보낼 때 바닥에 쏟아두면 다른 빨래와 섞인다. 단계 사이에 임시 보관 박스를 두고 “어느 작업의 데이터인지”라는 꼬리표를 함께 담아 다음 공정으로 넘기는 구조가 pipeline register다.

TL;DR: Pipeline register는 명령어들이 꼬이지 않도록 정보의 독립성을 유지한다. 데이터뿐 아니라 control과 register 번호도 함께 전파한다.

6. Load 명령어의 datapath 오류와 수정

기본 pipeline에서 load 명령어가 write-back 단계에서 엉뚱한 register에 값을 기록할 수 있는 문제는, 목적지 register 정보의 도착 timing이 data와 동기화되지 않아서 발생한다.

치명적인 하드웨어 버그 분석

lw $10, 20($1)이 동작한다고 하자. 이 명령어는 WB stage에서 data memory에서 가져온 값을 register file의 $10에 써야 한다. 하지만 WB cycle이 되었을 때 register file의 Write Register 입력이 현재 ID stage에 있는 완전히 다른 최신 명령어의 register 번호를 바라본다면 문제가 생긴다.

결과적으로 lw가 가져온 data는 원래 가야 할 $10이 아니라 뒤쪽 instruction의 register에 덮어써질 수 있다. 이는 data는 pipeline을 타고 WB까지 왔지만, “어디에 써야 하는가”라는 destination identifier가 함께 오지 않았기 때문에 생기는 오류다.

하드웨어 해결책: Write register의 동반 전파

해결은 간단하지만 중요하다. ID stage에서 이미 확정된 목적지 register 번호, 즉 Rt 또는 Rd를 최종 WB stage까지 data와 함께 pipeline register에 태워 전파해야 한다. 그러면 register 번호는 ID/EX → EX/MEM → MEM/WB를 차례로 통과하고, WB timing에 data와 정확히 일치된 상태로 register file의 write address에 들어간다.

직관적 비유: 주소지가 찢어진 택배

  • 오류 상황: 택배 허브에서 목적지 주소를 확인했지만, 배송 상자만 트럭에 싣고 주소 송장은 두고 간 상황이다. 나중에 배송 기사는 엉뚱한 최신 주소를 보고 오배송한다.
  • 정상 해결: 주소 송장 라벨을 박스 옆면에 붙여서 배송 과정 끝까지 함께 이동시킨다. Pipeline에서는 write register number를 data와 함께 전달하는 것이다.

TL;DR: Load bug의 원인은 data와 write destination의 cycle 불일치다. 목적지 register 주소 정보 역시 pipeline register를 통해 WB까지 동반 전파해야 한다.

7. 파이프라인 제어 신호의 전파

Control signal은 ID stage에서 한 번에 생성되고, 해당 signal이 사용될 물리 stage에 맞춰 pipeline register를 타고 instruction과 동기화되어 흐른다.

중앙 통제실이 필요 없는 동기화 제어

Pipeline 공장에서는 매 cycle마다 다른 조립을 지시할 초고성능 중앙 통제 장치가 필요하지 않다. 대신 ID stage에서 opcode를 보고 필요한 단계별 control line 값을 미리 결정한다. 그 후 signal bundle을 pipeline register의 일부로 넣어 data와 동일하게 한 칸씩 옆으로 밀어 전파한다.

Exam answer skeleton

  1. Single-cycle의 critical path 문제를 정의한다.
  2. Pipeline이 latency가 아니라 throughput을 개선한다는 점을 밝힌다.
  3. IF/ID/EX/MEM/WB와 pipeline register를 연결한다.
  4. Load bug는 destination register 정보 보존 문제로 설명한다.
  5. Control signal은 ID에서 생성되어 stage별로 전파된다고 정리한다.

Do not confuse

Pipeline의 이상적 CPI≈1은 한 명령어가 1 cycle만에 끝난다는 뜻이 아니다. Pipeline fill 이후 매 cycle마다 하나의 instruction이 완료될 수 있다는 throughput 설명이다.

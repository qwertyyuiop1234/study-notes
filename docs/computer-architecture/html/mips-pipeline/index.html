1. Single Cycle CPU의 한계와 파이프라인 개요

"싱글 사이클 구현은 모든 명령어가 동일한 1클럭 주기를 가지므로, 가장 긴 경로(Critical Path)인 Load 명령어에 의해 전체 클럭 주기가 강제 하향 평준화되는 비효율성을 갖습니다."
싱글 사이클 CPU의 작동 한계점

싱글 사이클(Single Cycle) 하드웨어 설계는 아주 단순하고 이해하기 쉽습니다. 하나의 클럭 주기 동안 단 하나의 명령어만 완전히 시작되고 완료됩니다. 하지만 이 철학은 치명적인 성능 저하의 주범입니다. CPU 클럭 주기의 길이는 가장 긴 지연(Delay)을 갖는 명령어에 맞춰 결정되어야 합니다. MIPS 명령어 중 가장 긴 작업 경로인 임계 경로(Critical Path)를 갖는 것은 바로 Load 명령어(lw)입니다. lw는 명령어 메모리 읽기 → 레지스터 파일 읽기 → ALU 주소 연산 → 데이터 메모리 읽기 → 레지스터 파일 다시 쓰기 단계를 모두 차례대로 통과해야 하므로 엄청난 전파 지연을 수반합니다.

공통 케이스의 고속화 (Making the Common Case Fast) 원칙 위배

컴퓨터 아키텍처의 황금 규칙인 "가장 흔하게 발생하는 일을 빠르게 만들라"는 원칙에 비추어 볼 때, 매우 드물게 쓰이는 무거운 명령어 때문에 자주 실행되는 단순 연산(예: 덧셈, 분기문)의 속도까지 강제로 낮춰야 하는 싱글 사이클 구조는 비효율적입니다. 명령어에 따라 유동적으로 클럭의 주기를 매번 조절하는 것은 물리적 회로 설계 수준에서 불가능에 가깝습니다. 이 성능 문제를 완전히 격파하기 위해 도입된 혁신이 바로 파이프라이닝(Pipelining)입니다.

직관적 비유: 대형 마트의 무식한 계산대

물건 하나를 산 사람과 카트 가득 물건을 채운 사람이 뒤섞여 있는 계산대가 있습니다. 싱글 사이클은 카트 가득 채운 사람의 긴 계산이 완벽하게 끝나서 영수증을 받고 매장 문을 나설 때까지, 뒤에 대기하는 물건 1개만 가진 사람의 결제를 단 1초도 진행하지 않고 대기열을 통째로 멈춰놓는 답답한 계산대와 같습니다.

🤔 그렇다면 명령어가 끝날 때마다 하드웨어가 스스로 눈치를 보며 클럭 속도를 높였다 낮췄다 하면 안 되나요?
물리적인 클럭 신호 발생기(Oscillator)와 회로의 동기화 장치는 일정한 주기로 파동을 치도록 설계됩니다. 클럭 주기를 매 명령어의 동적 실행 경로에 따라 아주 미세한 피코초(ps) 단위로 늘였다가 줄이는 제어 회로를 설계하는 것은 엄청난 회로 복잡도와 연산 지연을 초래하므로 절대 구현 불가능합니다.
TL;DR / 핵심 요약

- 임계 경로: 가장 긴 지연을 가지는 lw 명령어가 싱글 사이클의 최대 클럭 스피드를 제한함.
- 해결책: 클럭 주기를 쪼개 여러 명령어를 겹쳐 수행하는 파이프라이닝의 도입.

2. 빨래방 비유로 이해하는 파이프라인 법칙

"파이프라이닝은 개별 작업의 소요 시간(Latency)을 단축시키지 않으며, 전체 작업 처리량(Throughput)을 비약적으로 증가시키는 기술입니다."
MIPS가 제시하는 빨래방 공식

네 사람(Ann, Brian, Cathy, Dave)이 각각 한 더미씩의 빨래를 세탁, 건조, 개기 순으로 처리해야 합니다. 각 공정별 소요 시간은 세탁기(Washer) 30분, 건조기(Dryer) 40분, 개기(Folder) 20분입니다.

순차적 빨래 vs 파이프라인 빨래 비교

- 순차적 빨래 (Sequential Laundry): Ann의 빨래가 세탁, 건조, 개기(총 90분)가 모두 끝난 후에야 비로소 Brian이 세탁기에 빨래를 넣습니다. 총 4개의 빨래 더미를 다 처리하려면 무려 6시간($90\text{분} \times 4 = 360\text{분}$)이 걸립니다.
- 파이프라인 빨래 (Pipelined Laundry): Ann의 세탁이 끝나고 건조기로 넘어가자마자, Brian은 기다리지 않고 즉시 세탁기를 돌립니다. 서로 다른 자원(세탁기, 건조기, 개기)을 동시에 사용해 공정을 겹치기 때문에, 4더미의 모든 빨래 처리가 단 3.5시간(210분) 만에 마무리됩니다!

구분 순차적 처리 (Sequential) 파이프라인 처리 (Pipelined)
총 소요 시간 (4더미) 6시간 (360분) 3.5시간 (210분)
개별 Latency (1더미) 90분 (단축되지 않음) 90분 (동일함, 오히려 레지스터 오버헤드로 아주 미세하게 늘어날 수도 있음)
시간당 처리량 (Throughput) 0.66 더미/시간 1.14 더미/시간 (약 1.7배 상승!)
파이프라인이 주는 위대한 컴퓨터 아키텍처적 교훈

1. 개별 응답시간(Latency)은 감소하지 않는다: Ann의 빨래는 여전히 끝날 때까지 90분이 걸립니다. 단지 전체 일처리 효율이 올라갈 뿐입니다.
2. 속도 향상비(Speedup)의 한계는 파이프라인 단계 수와 일치한다: 이론상 $N$단계 파이프라인을 구축하면 성능은 최대 $N$배로 향상됩니다.
3. 가장 느린 단계(Slowest Stage)가 전체 속도를 좌우한다: 건조기가 40분으로 가장 느리기 때문에, 세탁이 아무리 30분 만에 끝나도 다음 단계인 건조기 앞에 정체 현상이 빚어집니다. 따라서 파이프라인 효율을 극대화하려면 각 단계의 일처리 시간이 완벽하게 균등(Balanced)해야 합니다.

TL;DR / 핵심 요약

- 파이프라인은 개별 명령어의 실행 시간(Latency)이 아니라, 단위 시간당 끝마치는 명령어 수(Throughput)를 증가시킵니다.
- 가장 느린 공정(Bottleneck)이 전체 시스템 스피드의 한계를 규정합니다.

3. MIPS 5단계 파이프라인 (IF-ID-EX-MEM-WB)

"MIPS 구조는 명령어를 실행하는 모든 과정을 완벽히 독립된 5개의 물리적 단계(Stage)로 잘게 쪼개어 동시 실행 흐름을 제어합니다."
공장을 구성하는 5개의 핵심 분업 라인

MIPS 구조에서 하나의 명령어가 동작하기 위해 반드시 순서대로 거쳐야만 하는 5개의 표준 물리 단계는 다음과 같이 철저하게 정의되어 있습니다.

1. IF (Instruction Fetch - 명령어 인출):
- PC(프로그램 카운터)가 가리키는 주소의 명령어 메모리(Instruction Memory)에서 32비트 기계어 명령어를 읽어옵니다. 이와 동시에 PC 값을 4만큼 증가시켜 다음 명령어가 위치한 주소를 가리키도록 설정합니다.

2. ID (Instruction Decode - 명령어 해석 & 레지스터 읽기):
- 들고 온 명령어가 무엇인지 분석하고(Decode), 동시에 필요한 피연산자가 담긴 레지스터 주소(Rs, Rt)에 접근하여 32비트 데이터를 읽어(Register Read)옵니다.

3. EX (Execute - 연산 또는 주소 계산):
- 핵심 연산 장치인 ALU를 기동시킵니다. R-type 연산이라면 레지스터 값을 이용해 연산하고, Load/Store라면 기저 주소에 오프셋 값을 더해 메모리 주소를 계산하며, Branch 연산이라면 비교 연산을 수행합니다.

4. MEM (Memory Access - 데이터 메모리 접근):
- 만약 메모리 작업 명령어(lw, sw)라면 계산된 데이터 메모리 주소 영역에 도달하여 실제 데이터 블록을 꺼내오거나(Load) 저장(Store)하는 데이터 물리 메모리 작업을 수행합니다. 그 외의 연산은 이 단계를 그냥 통과(Bypass)합니다.

5. WB (Write Back - 레지스터 쓰기):
- 최종 연산 완료값이나 데이터 메모리로부터 읽어온 로드 결과 데이터를 레지스터 파일(Register File)의 목적지 레지스터 주소 공간에 다시 온전하게 덮어씁니다.
직관적 비유: 대형 자동차 조립 공장 공정

- IF: 기본 뼈대 강판 가져오기
- ID: 엔진 및 조립 설명서 부품 꺼내기
- EX: 실린더와 프레임 용접하기
- MEM: 좌석 시트 및 유리창 가죽 내부 마감하기
- WB: 번호판 붙이고 완성차 등록하기
이 과정이 분업 컨베이어 벨트처럼 연결되어 끊임없이 완성차를 토출해 내는 흐름입니다.

TL;DR / 핵심 요약

MIPS의 5대 핵심 공정은 IF(가져오기) → ID(해석하기) → EX(계산하기) → MEM(메모리 읽기/쓰기) → WB(레지스터에 저장)로 빈틈없이 맞물려 전진합니다.

4. 파이프라인 성능 분석과 MIPS ISA의 비밀

"파이프라인의 1클럭 주기는 가장 느린 단일 단계의 지연 시간에 묶이지만, 균등하게 나누어진 물리 하드웨어는 클럭 사이클 시간을 급격히 단축시킵니다."
실제 수치(ps)로 분석하는 하드웨어 시간차

각 공정 소요 시간이 아래와 같다고 가정합니다:
- 레지스터 읽기/쓰기 공정 (ID, WB): 각각 100ps
- 나머지 공정 (IF, EX, MEM): 각각 200ps

• 싱글 사이클 구현 시 1클럭 주기: 가장 긴 lw 명령어 소요 시간인 800ps (200 + 100 + 200 + 200 + 100)가 주기 결정
• 파이프라인 구현 시 1클럭 주기: 가장 느린 공정 시간인 200ps (IF, EX, MEM의 소요 시간)가 주기 결정

100개의 명령어를 처리할 때의 극적인 격차 유도

1) Single Cycle Machine:
   Cycle time = 45ns, CPI = 1
   총 소요 시간 = $45\text{ ns} \times 1\text{ CPI} \times 100\text{ inst} = \mathbf{4500\text{ ns}}$

2) Multicycle Machine:
   Cycle time = 10ns, 평균 CPI = 3.6
   총 소요 시간 = $10\text{ ns} \times 3.6\text{ CPI} \times 100\text{ inst} = \mathbf{3600\text{ ns}}$

3) Ideal Pipelined Machine:
   Cycle time = 10ns, CPI = 1 (단, 파이프라인을 완전히 채우고 빠져나오는 시간 4사이클 감안)
   총 소요 시간 = $10\text{ ns} \times (100\text{ inst} + 4\text{ cycles drain}) = \mathbf{1040\text{ ns}}$

• 성능 개선비: $4500\text{ ns} / 1040\text{ ns} = \mathbf{4.33\text{배}}$ 속도 향상 달성!

MIPS 명령어 세트(ISA)가 파이프라인에 최적화된 이유

MIPS 구조는 탄생 초기부터 파이프라이닝을 염두에 두고 철저하게 하드웨어가 설계되었습니다.

  • 모든 명령어가 32비트 고정 길이: 명령어 크기가 일정하므로, 1단계(IF)에서 명령어를 가져오는 일과 디코딩하는 일이 극도로 일관되게 끝납니다. (가변 길이를 쓰는 x86 설계는 명령어 패치 자체부터 극심한 오버헤드를 수반합니다.)
  • 정형화되고 단순한 명령어 형식: 레지스터 번호가 항상 고정된 비트 위치에 존재하므로, 기계어가 디코더에 들어오는 2단계(ID)와 동시에 레지스터 데이터를 먼저 병렬적으로 읽어낼 수 있습니다.
  • 엄격한 Load/Store 아키텍처: 오직 lw와 sw 명령어로만 메모리에 도달할 수 있으므로, 3단계(EX)에서 주소를 산출하고 곧장 4단계(MEM)에서 데이터 메모리를 밟는 규칙성이 철저하게 보장됩니다.
TL;DR / 핵심 요약

- 파이프라인은 클럭 스피드를 올려 극적인 속도 성능(예: 4.33배)을 유도합니다.
- 32비트 고정형 기계어 포맷과 단순한 레지스터 배치 등 MIPS ISA 설계 구조가 하드웨어 분업 공정의 복잡도를 혁신적으로 줄였습니다.

5. 파이프라인 레지스터(Registers)와 정보 격리

"파이프라인 레지스터는 각기 다른 사이클에서 처리 중인 서로 다른 명령어들이 서로의 연산 정보를 덮어쓰지 못하도록 단계 사이를 물리적으로 격리하는 체크포인트 장벽입니다."
단계별 격리벽: 파이프라인 레지스터

동시에 5개의 명령어가 파이프라인 안에서 사이 좋게 흐르고 있습니다. 만약 단계 사이에 어떠한 임시 저장 장치도 없다면, 3단계(EX)에서 ALU 연산을 수행하는 순간 뒤따라오던 2단계(ID)의 명령어가 던진 입력값이 혼선을 빚어 연산 장치를 무작위로 망쳐버릴 것입니다. 따라서 각 5대 공정 단계 경계면에는 매 클럭마다 이전 결과를 딱 잡아두는 파이프라인 레지스터가 의무적으로 배치되어야만 합니다. 이 레지스터들은 시스템 주 클럭 신호에 맞추어 일괄적으로 정보를 다음 단계로만 밀어줍니다.

파이프라인 레지스터 비트 폭의 과학적 계산법

각 단계 사이에 배치된 파이프라인 레지스터들은 다음 단계가 필요로 하는 모든 데이터와 지시사항을 안전하게 넘겨주기 위해 계산된 충분한 크기(비트 폭)를 갖습니다.

  • IF/ID 레지스터 (64 bits wide):
    - 가져온 명령어 기계어 코드(32 bits) + 4 증가된 PC 주소값(32 bits) = 총 64 bits
  • ID/EX 레지스터 (147 bits wide):
    - 제어 신호 패키지(9 bits) + 읽어온 레지스터 데이터 1(32 bits) + 읽어온 레지스터 데이터 2(32 bits) + 32비트로 확장된 부호확장 상숫값(32 bits) + PC+4 주소(32 bits) + Rt 레지스터 번호(5 bits) + Rd 레지스터 번호(5 bits) 등 = 총 147 bits
  • EX/MEM 레지스터 (107 bits wide):
    - 남은 제어 신호(5 bits) + 분기 타겟 주소(32 bits) + ALU 연산 결과(32 bits) + ALU Zero 플래그(1 bit) + Write할 레지스터 데이터 2(32 bits) + 최종 Write Register 번호(5 bits) = 총 107 bits
  • MEM/WB 레지스터 (71 bits wide):
    - 최종 제어 신호(2 bits) + 데이터 메모리로부터 읽어온 데이터(32 bits) + ALU 연산 결과값(32 bits) + 최종 Write Register 번호(5 bits) = 총 71 bits
직관적 비유: 공장의 조립식 서랍 보관함

세탁이 완료된 축축한 옷들을 다음 공정으로 보낼 때, 그냥 바닥에 무작위로 쏟아버리면 다른 빨래들과 전부 뒤섞여 대참사가 벌어집니다.
단계 사이에 임시 보관 박스(파이프라인 레지스터)를 하나씩 두고, "Ann의 세탁물"이라는 꼬리표 정보(Control & Data)를 서랍 안에 통째로 담아서 다음 공정 담당자에게 질서정연하게 넘겨주는 구조와 완벽히 동일합니다.

TL;DR / 핵심 요약

- 파이프라인 레지스터는 명령어들이 꼬이지 않도록 정보의 독립성을 격리 유지하는 벽입니다.
- IF/ID(64) → ID/EX(147) → EX/MEM(107) → MEM/WB(71) 크기로 데이터가 전파됩니다.

6. Load 명령어의 치명적 설계 오류와 수정

"기본 파이프라인의 Write-Back 단계에서 Load 명령어가 엉뚱한 레지스터에 연산값을 기록하는 버그는 목적지 레지스터 정보의 소유 타이밍이 동기화되지 않아 발생합니다."
치명적인 하드웨어 버그 분석

기본 MIPS 파이프라인 설계도 상에서 lw $10, 20($1) 명령어가 동작한다고 가정해봅시다. 이 명령어는 최종 5단계(WB)가 되었을 때, 데이터 메모리에서 꺼내온 소중한 값을 Register File의 $10번 레지스터에 써주어야 합니다. 하지만 5번째 사이클이 도달한 시점에, Register File의 'Write Register 번호' 입력 포트는 무엇을 바라보고 있을까요? 그 시점에 2단계(ID)를 밟으며 새로 들어온 완전히 무관한 최신 명령어의 목적지 레지스터 주소 정보를 바라보게 됩니다! 결국 lw가 가져온 데이터는 원래 가야 할 $10번이 아닌, 엉뚱한 뒤쪽 명령어의 레지스터 메모리에 덮어써져 프로그램이 완전히 파괴되는 참극이 발생합니다. (강의 자료 "Wrong register number" 슬라이드 분석)

우아한 하드웨어 해결책: Write Register의 동반 전파

이 문제를 타파하는 하드웨어 솔루션은 간단하지만 천재적입니다. 2단계(ID)에서 명령어가 해석될 때, 이미 확정된 목적지 레지스터의 번호(Rt 혹은 Rd)를 메모리 최종 WB 단계까지 데이터와 함께 파이프라인 레지스터에 태워 함께 전파하는 것입니다! 이로써 레지스터 번호는 ID/EX → EX/MEM → MEM/WB 레지스터를 차례로 안전하게 통과한 후, 5단계(WB) 사이클 타이밍에 정확하게 데이터와 일치된 상태로 Register File의 Write 주소선에 꽂혀 정상 동작합니다.

직관적 비유: 주소지가 찢어진 택배 화물

- 오류 상황: 택배 허브(ID)에서 목적지 주소 정보($10번 레지스터)를 확인해놓고, 배송 상자만 트럭에 실어 보내고 정작 중요한 주소지는 본사에 그냥 놔둔 상황입니다. 배송 기사가 5일 뒤 도착했을 때 주소가 없으므로 본사에 물어봤더니, 현재 본사에 막 접수된 엉뚱한 다른 사람의 주소지로 오배송 처리가 납니다.
- 정상 해결: 주소 송장 라벨(Write Register 번호)을 박스 옆면에 견고하게 붙여서(파이프라인 레지스터에 기록) 택배와 완벽하게 끝까지 같이 동행하도록 만든 것입니다.

TL;DR / 핵심 요약

- 버그의 원인: 데이터와 쓰기 대상 주소의 도착 사이클 불일치
- 해결책: 목적지 레지스터 주소 정보 역시 파이프라인 레지스터를 통해 최종 WB 단계까지 함께 동반 전파시킴.

7. 파이프라인 제어 신호(Control Signals)의 전파

"제어 신호는 명령어 해석(ID) 단계에서 단 한 번에 생성되며, 해당 신호가 사용될 물리 단계에 맞춰 파이프라인 레지스터를 타고 명령어와 동기화되어 흘러갑니다."
중앙 통제실이 필요 없는 동기화 제어 메커니즘

파이프라인 공장에서는 매 사이클마다 다른 조립을 지시할 초고성능 중앙 통제 장치가 불필요합니다. 대신 명령어 해석 단계(ID)에서 이미 명령어의 앞머리(Opcode)를 보고 필요한 모든 단계별 제어 라인의 전선 전압(High/Low) 값을 싹 결정해버립니다. 그 후, 이 신호 뭉치들을 파이프라인 레지스터의 한 영역에 가지런히 적재하여 데이터와 완전히 동일하게 한 칸씩 옆으로 밀어 전파시킵니다.

공정 단계별 사용되는 제어선 매핑 설계

제어 신호는 자신이 실제로 필요한 공정 단계를 밟을 때 비로소 장치 제어선에 직접 흘러 들어갑니다:

  • EX 단계용 제어 신호 (ID/EX에서 즉각 소모): RegDst (목적지 선택), ALUOp1, ALUOp0 (ALU 종류), ALUSrc (피연산자 선택)
  • MEM 단계용 제어 신호 (EX/MEM 레지스터에 실려 전파): Branch (분기 판별), MemRead (메모리 읽기), MemWrite (메모리 쓰기)
  • WB 단계용 제어 신호 (MEM/WB 레지스터까지 흘러 전파): RegWrite (레지스터 기록 여부), MemtoReg (기록용 데이터 소스 선택)

TL;DR / 핵심 요약

- 제어 신호들은 ID 단계에서 미리 일괄 생성된 후, 데이터패스를 따라 이동하며 각 공정 시점에 순차적으로 소모됩니다.
- 이 흘려 보내는 성질 덕분에 하드웨어가 복잡한 제어 상태를 고민할 필요 없이 극도의 규칙성을 유지하게 됩니다.

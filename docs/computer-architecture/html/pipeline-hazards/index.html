2. 수식과 정의

Chapter 3

Forward Unit 조건문 해부

Forwarding은 ALU input 앞에 mux를 추가하고, ForwardA, ForwardB control signal로 어느 값을 사용할지 선택한다.

00

Normal input: ID/EX pipeline register 값 사용.

10

EX/MEM에서 forwarding. 바로 이전 instruction 결과.

01

MEM/WB에서 forwarding. 두 instruction 앞 결과.

EX/MEM hazard 조건문

if (EX/MEM.RegWrite
    and EX/MEM.RegisterRd != 0
    and EX/MEM.RegisterRd == ID/EX.RegisterRs)
    ForwardA = 10;

if (EX/MEM.RegWrite
    and EX/MEM.RegisterRd != 0
    and EX/MEM.RegisterRd == ID/EX.RegisterRt)
    ForwardB = 10;

MEM/WB hazard와 최신 결과 우선순위 (Priority logic)

MEM/WB도 source register와 destination register가 같으면 forwarding 후보가 된다. 그러나 EX/MEM과 MEM/WB가 동시에 match되면 EX/MEM이 더 최신 결과이므로 우선해야 한다.

if (MEM/WB.RegWrite
    and MEM/WB.RegisterRd != 0
    and MEM/WB.RegisterRd == ID/EX.RegisterRs
    and not(EX/MEM.RegWrite
            and EX/MEM.RegisterRd != 0
            and EX/MEM.RegisterRd == ID/EX.RegisterRs))
    ForwardA = 01;
📌 시험 포인트: 조건문 서술 시에는 단순 register match뿐 아니라 RegWrite = 1 여부, RegisterRd != 0(제공자가 $zero가 아닐 것) 조건, 그리고 최신 결과를 먼저 가져오는 우선순위 로직(not EX/MEM match)이 반드시 명시되어야 만점을 받습니다.

3. 직관과 그림

Chapter 0

파이프라인의 그림자, 해저드란 무엇인가?

Pipeline hazard는 다음 clock cycle에 다음 instruction을 정상적으로 시작하지 못하게 만드는 hardware 또는 dependency 상황이다. Pipeline은 여러 instruction을 동시에 겹쳐 실행하기 때문에, 값이 “논리적으로는 앞에서 만들어져야 한다”는 사실만으로 충분하지 않다. 실제 hardware stage에서 그 값이나 resource가 준비되어 있어야 한다.

Structural

같은 cycle에 두 instruction이 같은 물리 resource를 서로 다른 방식으로 요구하는 경우. 대표 예시는 single memory conflict.

Data

이전 instruction 결과가 아직 준비되지 않았는데 다음 instruction이 그 register 값을 읽거나 ALU input으로 사용하려는 경우.

Control

branch condition과 target이 확정되기 전에 다음 PC를 결정해야 하는 경우. 이 노트는 주로 data hazard 해결에 집중한다.

TL;DR: 모든 hazard는 최악의 경우 기다리면 해결된다. 하지만 기다림은 CPI를 증가시키므로, forwarding과 scheduling으로 최대한 bubble을 줄이는 것이 핵심이다.

4. 예제

Chapter 5

Code Scheduling: 빈 cycle을 유용한 instruction으로 채우기

Compiler는 program semantics를 보존하면서 independent instruction을 load-use gap에 끼워 넣어 stall을 줄일 수 있다.

스케줄링 전 (Stall 2개 발생)

lw  $t1, 0($t0)    # B
lw  $t2, 4($t0)    # E
stall
add $t3, $t1, $t2  # A = B + E
sw  $t3, 12($t0)
lw  $t4, 8($t0)    # F
stall
add $t5, $t1, $t4  # C = B + F
sw  $t5, 16($t0)
# 총 13 cycles

스케줄링 후 (Stall 완전 제거)

lw  $t1, 0($t0)    # B
lw  $t2, 4($t0)    # E
lw  $t4, 8($t0)    # F (위로 이동시켜 gap 채움!)
add $t3, $t1, $t2  # A = B + E
sw  $t3, 12($t0)
add $t5, $t1, $t4  # C = B + F
sw  $t5, 16($t0)
# 총 11 cycles

핵심은 load 직후 결과를 쓰는 instruction 사이에 의존성이 없는 작업을 배치하는 것이다. “순서를 바꿔도 같은 의미인가?”와 “load-use gap을 메웠는가?”를 동시에 확인해야 한다.

5. 자주 틀리는 부분

Chapter 1

Structural Hazard: 단일 메모리의 병목

Structural hazard는 resource conflict다. 5-stage MIPS pipeline에서 IF stage는 매 cycle instruction memory를 읽고, MEM stage의 lw/sw는 data memory를 읽거나 쓴다. 두 memory가 하나의 physical memory라면 IF와 MEM이 동시에 memory port를 요구하는 cycle에서 conflict가 생긴다.

해결책 효과 Trade-off
Stall / bubble Instruction fetch를 잠시 멈춰 correctness 보장 Throughput 감소, CPI 증가
Instruction/Data memory 분리 IF와 MEM의 동시 접근 허용 (Harvard Architecture) Cache/Memory 구조와 비용 증가
Multi-port memory 한 cycle에 복수 접근 허용 회로 복잡도와 면적 증가

Register file은 왜 덜 문제가 되는가?

MIPS-style pipeline은 흔히 register write를 cycle의 first half, register read를 second half에 수행하는 split-phase timing을 가정한다. 그래서 WB stage가 같은 cycle 앞부분에 write한 값을 ID stage가 뒤부분에 읽을 수 있어 hazard 확률을 절반으로 낮춘다.

Chapter 2

Data Hazard: RAW 의존성과 시간 역행 화살표

Data hazard의 대표는 RAW(Read After Write)이다. 뒤 instruction이 읽어야 할 값이 앞 instruction의 write 이후에야 architectural register file에 반영되는 상황이다.

sub $2,  $1, $3   # $2 생성
and $12, $2, $5   # $2 즉시 사용: 위험!
or  $13, $6, $2   # $2 사용: 위험 가능
add $14, $2, $2   # timing에 따라 split-phase로 안전 가능
sw  $15, 100($2)  # base register로 $2 사용

단순 해결책은 compiler가 nop을 삽입하거나 hardware가 stall을 걸어 기다리는 것이다. 하지만 이 방식은 correctness는 보장해도 pipeline throughput을 떨어뜨린다. 그래서 hardware는 가능한 경우 forwarding을 사용한다.

Stall 기법

값이 register file에 write될 때까지 기다린다. 구현은 단순하지만 bubble이 발생하여 CPI 손실을 유발한다.

Forwarding / bypassing 기법

EX/MEM 또는 MEM/WB pipeline register의 임시 연산 결과를 ALU input으로 직접 Bypass 전달한다. 추가적인 Mux 회로가 필요하다.

Chapter 4

Load-use Hazard: 포워딩의 물리적 한계

lw 결과는 MEM stage 끝에서야 준비된다. 바로 다음 instruction이 EX stage에서 그 값을 필요로 하면, forwarding으로도 시간을 거슬러 값을 전달할 수 없다.

lw  $2, 20($1)    # loaded data is ready after MEM
and $4, $2, $5    # needs $2 at EX too early

따라서 load-use hazard는 1-cycle stall을 삽입하고, 그 다음 cycle에 forwarding으로 해결한다.

Hazard detection condition (ID stage에서 작동)

if (ID/EX.MemRead
    and ((ID/EX.RegisterRt == IF/ID.RegisterRs)
      or (ID/EX.RegisterRt == IF/ID.RegisterRt)))
    stall_the_pipeline();

Stall hardware가 실제로 하는 일 (Bubble 삽입)

  • PCWrite = 0: PC update를 막아 새 instruction fetch를 멈춘다.
  • IF/IDWrite = 0: ID stage instruction을 같은 위치에 유지한다 (freeze).
  • ID/EX control = 0: EX로 nop/bubble을 보낸다 (RegWrite=0, MemWrite=0을 넣어 side effect 완전 차단).
